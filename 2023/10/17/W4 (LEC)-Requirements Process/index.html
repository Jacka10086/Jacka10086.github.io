Requirements Process (需求过程)

需求过程是项目开发中至关重要的一部分，它有助于明确项目的目标和需求，以确保项目成功完成。以下是需求过程的关键步骤和详细解释：

Identify Stakeholders (识别利益相关者)

  • 确定项目中的各种利益相关者，包括客户、用户、管理层、开发团队等。
  • 利益相关者的识别有助于确保项目满足各方的期望和需求。

Elicitation (需求收集)

  • 进行需求收集活动，包括面对面会议、访谈、问卷调查等，以获取各方的需求和期望。
  • 需求收集是确保项目成功的关键，因为它有助于捕捉关键信息。

User Centered Design Principles (用户中心设计原则)

  • 采用用户中心的设计原则，将用户的需求和体验置于项目的核心。
  • 用户中心设计有助于创造用户友好的产品和服务，提高用户满意度。

Elaborate / Create User Stories (详细/创建用户故事)

  • 创造用户故事，这是描述系统功能的简短故事，通常从用户的角度编写。
  • 用户故事有助于明确项目的功能和需求，为开发团队提供清晰的方向。

Prioritise MoSCoW (优先级MoSCoW)

  • 使用MoSCoW法则（Must, Should, Could, Won’t）来确定需求的优先级。
  • 这有助于确保关键功能首先得到满足，同时管理项目范围。

Update / Refine New Information, ongoing feedback (更新/完善新信息，持续反馈)

  • 随着项目的推进，不断更新和完善需求，以反映新的信息和反馈。
  • 持续反馈是确保项目满足变化需求的关键。

Manage overall scope and project direction (管理总体范围和项目方向)

  • 管理项目的总体范围，确保项目保持在预定的目标和方向上。
  • 这有助于避免项目范围蔓延和失控。

Story Mapping (故事映射)

  • 使用故事映射技术将用户故事可视化，以更好地理解和管理项目需求。
  • 故事映射有助于创建项目路线图，明确功能的依赖关系。

Scope Wants vs Needs (范围的愿望与需求)

  • 区分项目中的愿望和需求，明确哪些是关键需求，哪些是可选愿望。
  • 这有助于优化资源分配和项目规划，确保满足核心需求。

以上是需求过程的关键步骤和详细解释，这些步骤有助于确保项目顺利进行并满足各方的期望。

Requirements Process (需求过程)

需求过程是项目管理和软件开发中的关键步骤，它有助于明确项目的目标和需求，确保项目的成功完成。以下是需求过程的各个关键步骤和详细解释：

Identify Stakeholders (识别利益相关者)

  • 识别项目中的各种利益相关者，包括客户、用户、管理层、开发团队等。
  • 利益相关者的识别有助于确保项目考虑到各方的需求和期望。

Elicitation (需求收集)

  • 进行需求收集活动，例如面对面会议、访谈、问卷调查等，以获取各方的需求和期望。
  • 需求收集是确保项目成功的关键，因为它有助于获取项目的关键信息。

User Centered Design Principles (用户中心设计原则)

  • 采用用户中心设计原则，确保项目从用户的角度出发，提供出色的用户体验。
  • 用户中心设计有助于创建符合用户期望的产品和服务。

Elaborate / Create User Stories (详细/创建用户故事)

  • 创建用户故事，这些是简短的描述系统功能的故事，通常从用户的角度编写。
  • 用户故事有助于明确项目的功能和需求，为开发团队提供清晰的方向。

Prioritise MoSCoW (MoSCoW 优先级)

  • 使用MoSCoW法则（Must, Should, Could, Won’t）来确定需求的优先级。
  • 这有助于确保关键功能首先得到满足，同时管理项目范围。

Update / Refine New Information, ongoing feedback (更新/完善新信息，持续反馈)

  • 随着项目的推进，不断更新和完善需求，以反映新的信息和持续的反馈。
  • 持续反馈是确保项目满足变化需求的关键。

Manage overall scope and project direction (管理总体范围和项目方向)

  • 管理项目的总体范围，确保项目保持在预定的目标和方向上。
  • 这有助于避免项目范围蔓延和失控。

Story Mapping (故事映射)

  • 使用故事映射技术将用户故事可视化，以更好地理解和管理项目需求。
  • 故事映射有助于创建项目路线图，明确功能之间的依赖关系。

Scope Wants vs Needs (范围的愿望与需求)

  • 区分项目中的愿望和需求，明确哪些是必要需求，哪些是可选愿望。
  • 这有助于优化资源分配和项目规划，确保核心需求得到满足。

以上是需求过程的各个关键步骤和详细解释，它们共同确保项目按计划进行并满足各方的期望。

Requirements Engineering (需求工程)

需求工程是软件开发过程中至关重要的一部分，它有助于明确项目的需求，确保项目能够满足各方的期望。以下是与需求工程相关的关键信息：

  • Volere Template is popular (Volere模板很受欢迎)

    • Volere模板是一种流行的需求文档模板，用于捕捉和记录项目需求。
    • 这个模板通常包含了项目的各个方面，包括功能需求、非功能需求、用例等。

  • Easy to understand by all stakeholders (易于所有利益相关者理解)

    • 需求文档应该以简单明了的方式编写，以便所有利益相关者都能够理解其中的内容。
    • 清晰的需求文档有助于避免误解和沟通问题。

  • Based on a template (基于模板)

    • 创建需求文档时，可以使用事先定义的模板，如Volere模板，以确保文档的一致性和完整性。
    • 模板提供了一种结构化的方法来组织需求信息。

  • Functional Requirements (功能需求)

    • 功能需求描述了系统需要执行的各种功能。
    • 这些需求通常包括用户的操作、系统的响应和功能之间的关系。

  • Atomic (原子性)

    • 需求应该具有原子性，即它们应该是不可分割的单元。
    • 原子性需求更容易管理和测试。

  • Measurable (可测量性)

    • 需求应该是可测量的，这意味着可以定义明确的标准来验证需求是否已满足。
    • 可测量性有助于确定项目进展和成功标准。

  • Detailed (详细性)

    • 需求应该足够详细，以便开发团队明确了解需要实现的内容。
    • 不充分详细的需求可能导致误解和不完整的实现。

  • Testable (可测试性)

    • 需求应该是可测试的，以便能够创建测试用例来验证其实现。
    • 可测试性有助于确保系统符合规范。

  • Traceable (可追溯性)

    • 需求应该是可追溯的，即可以跟踪到其源头，并追踪其变化历史。
    • 可追溯性有助于维护需求的一致性和可管理性。

需求工程是项目成功的关键组成部分，确保需求的清晰性、可追踪性和可测试性对于项目的顺利进行至关重要。

INVEST in Good User Stories

在创建用户故事时，遵循INVEST原则可以帮助确保用户故事质量高，易于管理和开发。以下是INVEST原则的各个方面：

  • Independent (独立性)

    • 用户故事应该是自包含的，以减少对其他用户故事的依赖性。
    • 这有助于并行开发和更好的任务分配。

  • Negotiable (可协商性)

    • 用户故事应该关注“做什么”，而不是“如何做”。
    • 不应该过于具体，以便在开发过程中可以进行协商和调整，而不限定了解决方案。

  • Valuable (有价值性)

    • 用户故事必须包含明确的用户价值，以便在实现后带来实际好处。
    • 用户故事应该解决实际问题或满足用户需求。

  • Estimated (可估算性)

    • 队伍应该能够相对估算用户故事的大小。
    • 这有助于计划和资源分配。

  • Small (小型)

    • 用户故事应该足够小，以便在1到2个团队工作日内（不超过1/4个迭代）完成。
    • 较小的用户故事更易于管理和测试。

  • Testable (可测试性)

    • 用户故事应该提供明确的完成定义或验收标准，以便可以进行测试。
    • 这有助于确保用户故事在实现后能够验证其正确性。

遵循INVEST原则有助于创建高质量的用户故事，这些用户故事易于理解、开发和测试，从而提高了整体项目的效率和质量。

User Story Format (用户故事格式)

用户故事是敏捷开发中用于描述需求的简洁方式。以下是标准的用户故事格式：

  • Narrative (故事描述)

    • 用户故事通常以一段简短的故事描述开始，用于提供上下文和背景信息。

  • As a <type of user> (作为<用户类型>)

    • 指明故事的用户类型，即谁将受益于这个用户故事。

  • I want to <do something / have something> (我想要<做某事/拥有某物>)

    • 描述用户希望完成的操作或需要拥有的东西。

  • So that <for some benefit / value is created> (以便<为了某种利益/创造价值>)

    • 解释用户完成上述操作或拥有上述物品的目的，即为了获得何种价值或好处。

  • Acceptance Criteria (验收标准/完成定义)

    • 提供明确的验收标准或定义完成的标准，以确保用户故事在实现后可以被验证。
    • 验收标准通常包括具体的行为、条件和结果，以便明确了解用户故事的预期工作方式。

使用这个标准的用户故事格式有助于团队更好地理解和记录需求，确保所有相关信息都被捕捉到，并帮助保持用户故事的一致性和清晰性。

User Stories (用户故事)

用户故事是描述需求的一种常用方式，以下是用户故事的基本结构和组成部分：

  • Title (标题)

    • 一句话描述一个活动或需求。

  • Narrative (故事描述)

    • 以一种故事性的方式描述用户故事，包括以下部分：
      • As a <role> (作为<角色>)
        • 指明哪个角色或用户类型将从这个用户故事中受益。
      • I want to <do something> (我想要<做某事>)
        • 描述用户希望完成的操作或需要满足的需求。
      • So that <benefit> (以便<获得某种好处>)
        • 解释用户完成上述操作或需求的目的，即为了获得何种好处或价值。

  • Acceptance Criteria (验收标准)

    • 用户故事的验收标准以场景的形式呈现，每个场景具有以下结构：
      • Scenario Title (场景标题)
        • 清晰区分不同场景的标题。
      • Given (前提条件)
        • 描述场景发生的上下文和前提条件。
        • 这里可以包括一些上下文信息。
      • When (事件)
        • 描述触发场景的事件或操作。
      • Then (结果)
        • 描述场景的预期结果或行为。
      • And (其他结果)
        • 可选项，用于描述场景的其他结果或行为。

使用这种结构化的用户故事格式，团队能够更好地理解、分析和验证需求，确保开发人员和利益相关者都在同一个页面上。

ATM User Story - Scenario 1 (ATM用户故事 - 场景1)

  • Title (标题)

    • 账户持有人取款

  • Narrative (故事描述)

    • 作为一个账户持有人，我希望能够从ATM机上取款，以便在银行关门时仍然可以获取现金。

  • Acceptance Criteria (验收标准)

Scenario 1: Account has sufficient funds (场景1：账户有足够的资金)

  • Given (前提条件)
    • 账户余额足以支付所请求的金额
    • 银行卡有效
    • ATM机内有足够的资金
  • When (事件)
    • 账户持有人请求取款
  • Then (结果)
    • ATM应该发放请求的金额
    • 账户余额应扣除发放的金额
    • 银行卡应该被归还

使用这个用户故事和场景描述，我们可以清晰地了解当账户持有人在账户有足够的资金时从ATM机上取款的过程，并且可以验证这个过程的正确性。

Good and Bad User Stories (好的和不好的用户故事)

用户故事是描述需求的一种方式，但它们可以是好的或不好的。以下是一些好的和不好的用户故事的示例，以及它们的评价：

不好的用户故事

  • 不好：作为用户，我想搜索导师（As a user, I want to search for tutors）

    • 这个用户故事过于模糊，没有具体说明是哪种用户类型需要搜索导师，也没有提供为什么需要搜索导师的理由。

  • 不好：作为家长，我想监视课程、与导师和学生通过私人渠道聊天，并记录会话（As a parent, I want to monitor lessons, chat with the tutor and pupil via private channels and to record sessions）

    • 这个用户故事包含多个功能，违反了单一职责原则，不够清晰，应该分解成更小的故事。

  • 不好：作为导师，我想管理支付的各个方面（As a tutor, I want to manage all aspects of payments）

    • 这个用户故事过于笼统，没有具体说明哪些方面的支付需要管理，缺乏细节和焦点。

好的用户故事

  • 好的：作为导师，我想查看我的每日日程，以便计划我的时间并准时参加课程（As a tutor, I want to see my daily schedule so I can plan my day and be on time for lessons）

    • 这个用户故事明确了用户类型（导师）、需求（查看每日日程）和目的（计划时间、准时上课），具有清晰的焦点。

  • 好的：作为学生，我想看到导师练习，以便更好地理解如何演奏（As a pupil, I want to see my tutor practicing so that I can better understand how to play）

    • 这个用户故事明确了用户类型（学生）、需求（看导师练习）和目的（更好地理解如何演奏），简洁明了。

以上示例说明了好的用户故事应该明确、具体、聚焦，而不好的用户故事可能过于模糊、复杂或含有多个功能点。

User Stories Are Vertical Slices (用户故事是垂直切片)

在敏捷开发中，用户故事通常被视为垂直切片，这意味着它们跨足了各个层次，包括多个层次的变更，以交付增量价值。

垂直切片 (Vertical Slices)

  • 垂直切片是指跨足了整个软件堆栈的变更。
  • 这意味着用户故事包括对每个架构层的更改，包括前端、后端、数据库等。
  • 垂直切片通常代表对系统的端到端功能或特性的完整实现。
  • 通过垂直切片，可以确保每个用户故事都提供了可用的、可验证的功能。

水平切片 (Horizontal Slices)

  • 与垂直切片不同，水平切片通常是在某个特定层面或领域上的变更，而不跨足整个堆栈。
  • 这可能包括对单一层（如前端或后端）的修改，而不考虑系统的完整性。
  • 水平切片通常无法单独提供可用的功能，因为它们可能需要其他切片的协作才能交付完整的价值。

包括各个架构层的变更

  • 用户故事的垂直切片包括对整个架构层的变更，确保整个系统的一致性和集成。
  • 这些变更可能涉及用户界面、业务逻辑、数据存储等方面的修改。

为交付增量价值而足够

  • 每个用户故事都应该足够具体，以便交付一个可用的增量，为用户提供实际的价值。
  • 这有助于确保用户故事是可验证的，并且在每个迭代中都能交付有意义的结果。

需要完成多个切片才能交付增量价值

  • 增量价值通常需要多个垂直切片的协作，以构建完整的功能或特性。
  • 这强调了团队合作和协调的重要性，以确保每个切片都按时交付，并且能够一起工作。

在敏捷开发中，用户故事的垂直切片是一种强调交付实际价值的方式，通过跨足整个系统来实现端到端的功能或特性。

Acceptance Criteria (验收标准)

验收标准是用户故事的重要组成部分，用于明确定义故事的边界、完成条件以及满足故事的需求。以下是好的验收标准的特点：

  • 好的验收标准 (Good AC):
    • 清晰地定义故事的边界，明确说明什么属于故事的范围，什么不属于范围之内。
    • 描述需要满足以完成故事的各种场景。
    • 使用直接、逻辑和简单的语言，以确保所有团队成员都能理解。
    • 针对每个场景，要么通过，要么失败。
    • 表达”是什么”而不是”如何做”，例如，”导师可以记录视频会话”而不是”导师点击按钮开始录制”。

好的验收标准是确保用户故事的质量和可验证性的关键。它们提供了对故事完成的清晰定义，帮助团队理解何时可以认为故事已经完成。

Formats of AC (验收标准的格式)

验收标准是用户故事的重要部分，它们用于明确说明何时可以认为用户故事已经完成。下面展示了两种不同的验收标准格式：

格式1：使用场景描述 (Scenario-Based)

用户故事： 作为用户，我希望能够登录我的帐户，以便查看和管理我的付款信息。

验收标准：

  • 场景1：用户登录 (User Log-in)
    • 给定用户位于登录页面时，
    • 当他们输入电子邮件地址和密码时，
    • 那么系统将允许用户登录系统。
  • 场景2：无效的登录 (Invalid Log-in)
    • 给定用户已经输入了他们的电子邮件地址和密码，
    • 当他们选择登录时，
    • 那么如果密码或电子邮件地址无效，系统将显示错误消息。

格式2：使用简单列表 (Simple List)

用户故事： 作为用户，我希望能够登录我的帐户，以便查看和管理我的付款信息。

验收标准：

  1. 可以看到登录选项。
  2. 可以输入电子邮件地址和密码。
  3. 只有在电子邮件地址和密码都有效的情况下才能登录。
  4. 如果电子邮件地址或密码无效，可以看到错误消息。

这两种格式都可以用来描述用户故事的验收标准，第一种基于场景描述，第二种使用简单的列表形式。选择哪种格式通常取决于团队的偏好和项目需求。不管选择哪种格式，验收标准的关键是清晰、可验证且明确地定义了完成条件。

Influences on Priority (影响优先级的因素)

在确定用户故事的优先级时，有多种因素可能会起作用。以下是一些常见的影响优先级的因素：

  • 价值 (Value)

    • 对项目目标产生最大影响的用户故事通常具有更高的优先级。
    • 如果一个用户故事的交付对于项目成功非常重要，那么它通常会被放在更高的位置。

  • 不交付 (Not Delivering)

    • 不交付某个用户故事可能会对项目的目标产生最大的负面影响。
    • 如果某个用户故事的未交付会对项目造成严重问题，那么它通常会被视为高优先级。

  • 反馈 (Feedback)

    • 用户故事应该是自包含的，以限制对其他用户故事的依赖。
    • 这可以确保可以尽早地获得用户反馈，并有助于提高优先级。

  • 难度 (Difficulty)

    • 用户故事的实施难度可能会影响其优先级。
    • 一些较容易实现的用户故事可能会被提前考虑，以快速交付价值。

  • 依赖关系 (Dependencies)

    • 有时，一个用户故事的实施可能取决于其他用户故事的完成。
    • 在这种情况下，必须考虑依赖关系，以确定哪个用户故事应该首先实施。

  • HiPPO (Highest Paid Person’s Opinion)

    • 高层管理或决策者的意见有时会影响用户故事的优先级。
    • 高薪职位人员（HiPPO）的看法可能会导致某些用户故事的提前交付。

这些因素可能会相互交织，影响用户故事的优先级。在决定优先级时，团队需要综合考虑这些因素，以确保最大程度地满足项目的需求和目标。

Story Mapping (故事地图)

故事地图是一种工具和技术，用于在项目中可视化用户故事和产品功能。它可以帮助团队更好地理解产品的整体结构和优先级，以及产品的演进过程。以下是故事地图的主要特点和用途：

  • 清晰地绘制MVP（最小可行产品）

    • 故事地图可以清晰地绘制出MVP所需的用户故事和功能，帮助团队了解最初需要交付的核心功能。

  • 识别差距和机会

    • 通过故事地图，团队可以识别产品中的差距和机会，以确定哪些功能需要添加或改进，以更好地满足用户需求。

  • 全局视角

    • 故事地图提供了产品的大局观，帮助团队看到产品的整体结构和愿景，而不仅仅是个别功能的集合。

  • 优先级排序

    • 故事地图有助于优先级排序，帮助团队确定哪些功能应该首先开发，以在产品中交付最大价值。

  • 持续地绘制产品旅程

    • 故事地图不仅仅是一次性的工具，它可以持续地演进，反映产品的变化和增长，以保持对产品方向的清晰认识。

通过使用故事地图，团队能够更好地规划、管理和交付产品，确保产品的发展方向与用户需求保持一致，并且在整个项目周期内保持清晰的视角。
