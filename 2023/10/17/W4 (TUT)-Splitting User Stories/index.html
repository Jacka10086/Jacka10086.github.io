为什么要拆分故事？ (Why Split Stories?)

1. 将其拆分为可管理的小块

  • 便于管理: 更小的故事意味着更易于理解和实施。
  • 提高效率: 团队可以快速完成小故事，并感受到成就感，从而提高团队的整体速度。

2. 足够小，能够在两周的冲刺中交付8-12个故事

  • 可预测的交付: 规划更准确，因为故事的规模较小，易于估算。
  • 持续的进步: 每个冲刺结束时都能看到实实在在的成果，有助于保持团队和利益相关者的动力。

3. 区分高价值和低价值

  • 优先级排序: 通过拆分，我们可以识别哪些故事具有最高价值，应该首先执行。
  • 减少浪费: 低价值的故事可以推迟或取消，从而集中精力在更有价值的工作上。

4. 产品待办事项自然处理优先级

  • 自然排序: 故事拆分后，可以更自然地根据其重要性和紧迫性在产品待办事项中排序。
  • 灵活的调整: 随着项目的进展或市场需求的变化，可以随时重新评估和调整优先级。

5. 较小的故事提高了生产力和团队动力

  • 快速反馈: 小故事使团队能够更快地完成工作并获得反馈，从而迅速学习和改进。
  • 提高士气: 完有成效的冲刺和项目交付可以提高团队的信心和士气。

6. 更好的资源规划 - 分配工作

  • 明确的责任: 通过故事拆分，团队成员可以清楚地了解他们的责任和期望。
  • 平衡的工作负载: 可以根据团队成员的能力和可用性更有效地分配任务。

拆分用户故事 (Splitting the User Story)

用户故事拆分过程可以分为几个步骤，每个步骤都对故事的大小、质量和可管理性有重要影响。

准备输入故事 (Prepare the Input Story)

  1. 确认故事内容: 确保用户故事清晰、完整，并且包含所有相关方的需求。
  2. 满足INVEST原则 (除了“小”之外): 故事需要是独立的(Independent)、可协商的(Negotiable)、有价值的(Valuable)、可估计的(Estimable)、小的(Small)、可测试的(Testable)，尽管在拆分前可能不完全满足“小”的条件。

应用拆分模式 (Apply the Splitting Patterns)

  1. 工作流步骤 (#1 Workflow Steps): 按照不同的工作流程步骤来拆分用户故事。
  2. 操作 (#2 Operations): 根据不同的操作或功能将大故事拆分为小故事。
  3. 业务规则变化 (#3 Business Rules Variations): 如果一个故事包含多种业务规则，可以考虑分别拆分。
  4. 其他… (#4 …): 根据需要，可以考虑其他拆分模式，如按照数据类型或错误条件等。

评估拆分 (Evaluate the Split)

  1. 80/20原则: 专注于大部分用户最关心的功能或需求，即20%的功能满足80%的需求。
  2. 等大小故事: 尽量使拆分后的小故事大小均等，这有助于更好的规划和估算。

通过这种方法拆分用户故事，我们可以更有效地管理产品待办事项，并确保团队专注于交付最大的价值。

INVEST原则在优质用户故事中的应用 (INVEST in Good User Stories)

INVEST原则是编写高质量用户故事的关键，每个字母代表一个重要的属性，这些属性有助于确保故事的清晰度和可管理性。

独立性 (Independent)

  • 限制依赖: 故事应该是自包含的，尽量减少对其他用户故事的依赖。
  • 自主排程: 独立性允许故事可以在不依赖其他故事完成的情况下被开发和测试。

可协商 (Negotiable)

  • 关注点在于什么而非如何: 故事描述了需要什么而不是怎么做，即不是由解决方案驱动的。
  • 灵活的细节: 故事的具体实现可以在故事的生命周期中与团队协商。

有价值 (Valuable)

  • 清晰的用户价值: 每个故事必须为用户或项目干系人提供明确的价值。
  • 价值导向: 故事的优先级是基于它们所提供的价值来决定的。

可估算 (Estimable)

  • 相对大小的估算: 团队应该能够估算故事的大小，即使是相对的，也足以进行规划。
  • 基于信息: 估算应该基于当前所知的信息，接受可能的调整。

小巧 (Small)

  • 规划与完成的大小: 故事需要足够小，能够在一到两个团队日内计划和完成（最多不超过1/4冲刺）。
  • 可管理的任务: 小故事使得跟踪和复查变得更简单，降低了复杂性。

可测试 (Testable)

  • 清晰的完成定义或验收标准: 每个故事必须有明确的条件，以确定它何时被视为完成。
  • 质量保证: 可测试性确保了故事的质量和最终的用户满意度。

遵循INVEST原则能够帮助团队创建出清晰、可实现且有价值的用户故事，从而提高项目的成功率。

#1 - 工作流步骤 (Workflow Steps)

作为内容管理者，我可以发布新闻故事到公司网站。工作流步骤包括编辑和法律审批，最后审核到暂存区，然后再到生产。

通过将一个大的用户故事拆分成基于不同工作流步骤的几个小故事，我们可以更细致地管理和执行每个步骤。

新的用户故事 (New User Stories):

  • 我可以直接发布新闻故事到公司网站。

    • 这个故事关注于直接发布功能，没有经过任何审批流程。

  • 我可以发布经过编辑审核的新闻故事。

    • 这个故事强调了编辑审核的重要性和需求。

  • 我可以发布经过法律审查的新闻故事。

    • 这个故事确保内容遵循法律指导原则和公司政策。

  • 我可以在暂存站点上查看新闻故事。

    • 在此故事中，内容管理者能够在一个安全的环境中预览故事，这是发布前的重要步骤。

  • 我可以从暂存区发布新闻故事到生产环境。

    • 这个故事描述了从暂存区到生产环境的最终发布步骤。

每个新的用户故事都代表了发布新闻到公司网站的不同步骤，使得流程更加清晰，也便于团队分配资源和时间来分别处理这些步骤。

#2 - 操作 (Operations)

作为用户，我可以管理我的账户。这里的“管理”涵盖了多个操作（CRUD - 创建、读取、更新、删除）。

通过将“管理账户”这一宏观功能拆分为几个具体操作的用户故事，可以更清晰地定义需求，同时使得开发和测试工作更具针对性和可管理性。

新的用户故事 (New User Stories):

  • 我可以注册一个账户

    • 此故事关注于用户创建新账户的能力，是用户管理生命周期的起点。

  • 我可以登录我的账户

    • 此故事允许用户通过验证身份来访问其账户，是日常用户体验的重要组成部分。

  • 我可以编辑我的账户设置

    • 此故事强调了用户自定义和更新个人信息的能力，提高了用户体验的个性化。

  • 我可以注销我的账户

    • 此故事提供给用户一个结束使用服务的选项，这在处理用户数据和隐私时是一个重要考量。

这样拆分用户故事后，每个故事都聚焦于账户管理的一个特定方面，使得需求更加明确，开发工作更加细化，也便于跟踪每个功能的实现和质量。

#3 - 商业规则变化 (Business Rule Variations)

作为用户，我可以用灵活的日期搜索航班。这里的“灵活的日期”揭示了潜在的商业规则，这些规则值得拥有自己的用户故事。

通过为不同的商业规则创建独立的用户故事，我们能更清晰地定义这些规则，并确保它们在产品开发过程中得到妥善处理。

新的用户故事 (New User Stories):

  • 我可以搜索“在x和y之间的n天”的航班

    • 此故事让用户能够定义一个具体的时间范围，并搜索在这个范围内的航班。

  • 我可以搜索“12月的周末”的航班

    • 此故事专注于用户寻找特定月份中周末的航班，满足对假期或特定时间的需求。

  • 我可以搜索“x和y的前后几天”的航班

    • 此故事允许用户在指定日期前后的几天内搜索航班，提供更多的灵活性和选择。

每个新的用户故事都针对“灵活的日期”这一功能的不同商业规则进行了细化，使得功能需求更加明确，并帮助团队理解和实现这些复杂的搜索条件。

#4 - 数据变化 (Variations in Data)

作为内容管理者，我可以创建新闻故事。在处理多语言内容时，我们考虑到数据的变化，特别是语言。

创建针对不同语言的独立用户故事，有助于确保每种语言的特定需求和差异都能得到充分考虑和处理。

新的用户故事 (New User Stories):

  • …用英语 (in English)

    • 此故事强调了内容管理者用英语创建新闻的能力，考虑到可能的地域差异和表达习惯。

  • …用日语 (in Japanese)

    • 此故事让内容管理者能够用日语撰写新闻，涵盖了特定的字符集和格式需求。

  • …用阿拉伯语 (in Arabic)

    • 此故事关注于使用阿拉伯语创建新闻的功能，特别注意从右到左的书写方向和其他语言特性。

每个新的用户故事都专注于一种特定的语言，从而允许团队细分需求，更好地处理多语言支持所带来的复杂性和特殊要求。

#5 - 数据输入方法 (Data Entry Methods)

作为用户，我可以搜索两个目的地之间的航班。数据输入的用户界面可以是简单的，也可以是更复杂的。

通过拆分用户故事，针对不同的数据输入方法（如简单输入字段或复杂的日历界面）创建独立的故事，可以帮助确保每种输入方式的特定需求都被充分理解和考虑。

拆分用户故事 (Splitting the User Stories):

  • …使用简单的日期输入字段 (using simple date input fields)

    • 此故事关注于基本的数据输入需求，允许用户通过简单的文本字段来输入日期，提供快速、无干扰的用户体验。

  • …使用一个花哨的日历界面 (with a fancy calendar UI)

    • 此故事提供了一个更复杂、更直观的日期选择过程，通过一个图形日历界面，用户可以更容易地浏览和选择日期。

这两个故事考虑了从最基本到更高级的用户界面需求，使团队能够根据项目的范围、时间和资源来优先考虑和实现不同的功能。

#6 - 主要影响 (Major Effect)

作为用户，我可以使用VISA、万事达卡(MasterCard)、美国运通卡(American Express)支付我的航班费用。很明显，主要的工作在于卡处理基础设施。支持额外的卡类型是微不足道的。

通过拆分用户故事，我们可以更明确地聚焦于支持不同类型的信用卡所需的工作量和基础设施需求。

拆分用户故事 (Splitting the User Stories):

  • …我可以使用一种信用卡类型支付 (I can pay with one credit card type - VISA, MasterCard, AMEX)

    • 此故事强调了支持单一信用卡支付的实现，是建立支付基础设施的第一步。

  • …我可以使用所有四种信用卡类型支付（假设已经实现了一种）(I can pay with all four credit card types, given one is already implemented)

    • 此故事考虑到扩展支付系统以接受多种信用卡的能力，前提是已经有一种信用卡支付方式被实现。

拆分的这两个故事允许团队逐步实施复杂的支付处理功能，从支持一种信用卡类型开始，然后扩展到更多的卡种，这样做可以更有效地管理开发工作，并确保质量和安全性。

#7 - 简单/复杂 (Simple/Complex)

作为用户，我可以搜索两个目的地之间的航班。当通过增加更多情景使故事变得更大时，故事的复杂性也就随之增加。

通过将用户故事分解为核心功能和额外的变化，我们可以确保团队集中精力在主要功能上，同时也能够灵活地扩展更多的用户需求。

拆分用户故事 (Splitting the User Stories):

  • …指定最大停靠次数 (specifying a max number of stops)

    • 此故事让用户能够限制搜索结果中的航班停靠次数，为想要更直接航线的用户提供便利。

  • …包括附近机场 (including nearby airports)

    • 此故事扩展了搜索功能，允许用户选择包括附近机场的航班，为用户提供更多的选择和灵活性。

  • …使用灵活日期 (using flexible dates)

    • 此故事增加了日期的灵活性，用户可以搜索在特定日期范围内的航班，而不是固定的出发和返回日期。

通过保持核心故事的简洁性，并将更多的变化拆分到它们自己的用户故事中，团队可以更有效地管理工作的优先级，并确保每个功能都得到充分的注意和实现。

#8 - 推迟性能优化 (Defer Performance)

作为用户，我可以搜索两个目的地之间的航班。有时，工作的难点在于使功能快速或安全 - 可以分为“使它工作”和“使它快速”。

在开发初期，重点可能是实现基本功能；但是，随着产品的发展，性能和安全性变得越来越重要。因此，通过拆分用户故事来分阶段处理这些关注点是很有意义的。

拆分用户故事 (Splitting the User Stories):

  • …慢 - 只是完成它并显示等待/搜索动画 (…slow - just get it done and display waiting/searching animation)

    • 此故事聚焦于实现基本的搜索功能，即使响应时间较长，用户也会看到一个等待或搜索中的动画，从而知道进程正在进行中。

  • …快（在5秒内完成） (…fast - in under 5 seconds)

    • 此故事的目标是优化已有功能的性能，确保搜索结果能在5秒内快速呈现给用户，提供更流畅的用户体验。

虽然推迟性能优化可以帮助团队快速推出产品并获得用户反馈，但也要注意不要累积过多的技术债务，应适时进行必要的重构和优化，确保产品的长期健康和可维护性。

#9 - 分解探索性任务 (Break out a Spike)

作为用户，我可以使用信用卡支付。当面临不确定性时，可以使用探索性任务（spike）来进行调查研究。

探索性任务是一种短期、集中的研究活动，旨在获取足够的信息和知识，从而能够准确地估计一个复杂任务的大小和复杂性。在这种情况下，它可以用来调查信用卡处理的不确定性和复杂性。

拆分用户故事 (Splitting the User Stories):

  • …调查信用卡处理过程 (…investigate credit card processing)

    • 此故事不涉及实际的实现，而是关注于收集关于信用卡处理方式、第三方服务商、安全性要求和合规性问题的信息。

  • …实施信用卡处理 (…implement credit card processing)

    • 在收集到必要信息并减少不确定性之后，此故事关注于实际的信用卡支付功能的开发和实施。

探索性任务是管理风险的有用方法，它们允许团队在承诺实施解决方案之前先了解和理解潜在的难题和挑战。我们将在下周更详细地讨论这个主题。

评估拆分 (Evaluate the Split)

在故事拆分后，对结果进行评估是至关重要的。这可以帮助我们确保故事的大小和范围更适合迭代开发，并且团队可以更有效地管理其工作负载和交付优先级。

1. 80/20 帕累托原则 (80/20 Pareto Principle)

  • 80%的工作在20%的时间内完成：换句话说，完成最后20%的工作需要80%的时间。这通常意味着大部分工作可以在相对较短的时间内完成，而细节和微调可能会花费大量时间。
  • 识别繁琐的部分：即可能价值较低的20%部分。选择揭示高价值与低价值的拆分方式，可以帮助团队集中精力在产生最大影响的工作上。

2. 选择可以得到更多大小相等的小故事的拆分方式 (Choose the Split That Gets You More Equally Sized Small Stories)

  • 更均等的故事大小：将一个8点的故事拆分成四个2点的故事比产生一个5点和一个3点的故事更有用。这有助于团队更有效地规划和分配工作，因为他们可以预期每个故事所需的工作量更一致。

通过这些评估标准，我们可以更好地决定如何拆分故事，以便它们更符合敏捷开发的需要，同时确保工作的价值最大化和工作流程的顺畅。

拆分过度的风险（When Is Too Far?）

故事拆分是为了提高敏捷团队的效率和响应能力，但过度拆分可能会产生反效果。识别过度拆分的迹象至关重要。

拆分过小的风险 (Risks of Breaking into Too Small Chunks)

  • 相互依赖：当故事拆分得过小，以至于它们不能独立于其他故事存在时，这可能会导致开发和测试工作的复杂性增加，因为每个小故事都变得依赖于其他故事。
  • 缺乏独立价值：如果拆分后的故事对用户来说没有单独的价值，这可能意味着故事已被拆分得过细，无法单独交付任何实质性结果。

团队自适应性 (Teams Will Find What Works Well for Them)

  • 适应和学习：每个团队都有其独特的需求和动态。通过实践和反思，团队将了解什么程度的拆分对他们最有效。
  • 持续改进：敏捷是关于持续改进的。团队应该定期评估他们的故事拆分策略，并进行必要的调整，以确保他们不是在拆分过度或不足。

通过警惕拆分过度的迹象，并允许团队找到适合自己的平衡点，我们可以确保故事拆分实践为敏捷开发带来积极的影响，而不是额外的负担。

水平切割与垂直切割（Horizontal vs. Vertical Slicing）

故事的切割可以是水平的也可以是垂直的，但两种方法有着根本的不同，这影响了用户收益的实现和故事的测试方式。

水平切割的缺点 (Disadvantages of Horizontal Slicing)

  • 用户无感知的收益：直到所有的水平故事完成，用户才能感知到益处。
  • 无法进行测试：所有故事完成之前，无法进行有效测试。
  • 架构集成问题隐蔽：直到所有故事完成，才能发现潜在的架构集成问题。
  • 错失高低价值切割机会：无法在故事之间区分高价值和低价值。

水平切割的可能性 (Possibility of Horizontal Slicing)

  • 故事可进行水平切割：尽管存在缺点，但某些故事仍可水平切割以变得更小。
  • 通常导致开发任务的识别：这种切割方式往往会揭示具体的开发任务。

垂直切割 (Vertical Slicing)

  • 仅影响UI层的故事：例如，对表单的可见更改，无需贯穿所有层。
  • 改变系统接口的故事：例如，一个API更新。
  • 非功能性的故事：更改可能是可见的，因为某些操作变得更快。
  • 更改数据库中的数据：其效果可能在以后某个时间点可见。

通过理解水平切割和垂直切割的不同，我们可以更好地制定故事，确保它们为用户提供价值，并在开发过程中保持可测试性和可管理性。
