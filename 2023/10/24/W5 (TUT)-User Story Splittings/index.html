Splitting User Stories (拆分用户故事)

Sequence of techniques (技术序列)

  • User goal decomposition (用户目标分解)
    • Split the user goals into sub goals (将用户目标分解为子目标)
      • As an owner of unwanted stuff, I can register for an online auction website so that I can sell my stuff.
      • As an owner of unwanted stuff, I can enter my registration details so that I can register for an online auction website.
      • As an owner of unwanted stuff, I can submit my registration details so that I can register for an online auction website.

总结: 用户目标分解是一种技术，将主要的用户目标拆分为更具体的子目标。

  • Three named goal levels (三个命名的目标层级)
    • underwater level (underwater level)
    • indigo (indigo)
    • subfunction level (subfunction level)
    • cloud/kite level (cloud/kite level)
    • white (white)
    • summary level (summary level)
    • sea level blue (sea level blue)
    • user goal level (user goal level)

总结: 根据特定的层级名称，可以将目标分解到不同的细节层级。

  • Split out NFRs (分拆非功能性需求)
    • Separate functional and non-functional requirements of a user story (将用户故事的功能性和非功能性需求分开)
    • Defer the following NFRs (推迟以下NFRs)
      • Performance - speed (性能 - 速度)
        • Scalability - large data or large set of users (可扩展性 - 大数据或大量用户)
      • Concurrency - Race conditions, data locking, concurrent users (并发性 - 竞争条件、数据锁定、并发用户)
      • Availability/Reliability - uptime, robustness (可用性/可靠性 - 运行时间、鲁棒性)
      • Security - protect from attack (安全性 - 防御攻击)
      • Usability / accessibility - easy to use for everyone (可用性/可达性 - 每个人都容易使用)
      • UX - making it look nice (用户体验 - 界面美观)
      • Cross browser / platform - work for a variety of clients (跨浏览器/平台 - 适用于多种客户端)
      • Internationalisation - language support (国际化 - 语言支持)

总结: 非功能性需求(NFRs)是描述系统如何工作的需求，这里列出了常见的NFRs并说明了某些可以推迟实施的NFRs。

  • Split by User Interface Channel (按用户界面通道拆分)
    • Special case for cross browser / platform support (跨浏览器/平台支持的特殊情况)
    • Multiple devices (多设备)
      • Desktop (桌面)
      • Tablet (平板)
      • Mobile (手机)
    • Platforms (平台)
      • Windows (Windows)
      • Mac (Mac)
      • iOS (iOS)
      • Android (Android)

总结: 根据不同的用户界面通道，如设备类型和操作平台，拆分用户故事。

  • Split by User Type (按用户类型拆分)
    • Some stories have a diverse range of users (某些故事有多种用户)
      • As a user I want to register
        • Teacher (老师)
        • Parent (家长)
        • Principle (校长)

总结: 考虑到不同的用户群体，可以按用户类型来拆分用户故事。

  • Split summary goals into user goals (将摘要目标拆分为用户目标)
    • As blogger I can manage my article (摘要层级)
    • User goal level (用户目标层级):
      • As a blogger I can create my article
      • As a blogger I can view my article
      • As a blogger I can edit my article
      • As a blogger I can delete my article

总结: 通过从摘要层级到用户目标层级的分解，我们可以更具体地定义用户的需求。

  • Split by Scenario / Flow (按场景/流程拆分)
    • Happy paths versus unhappy paths (正常路径对比异常路径)
    • Deliver common happy paths (交付常见的正常路径)
    • Identify alternative flows (识别替代流程)
      • Sign in
        • success
        • Invalid username or password
        • failed 3 times
        • account locked

总结: 通过考虑正常和异常的场景或流程，我们可以更全面地描述用户故事。

  • Bronze plated / Gold plated flow (青铜版/金版流程)
    • As blogger I want to register
      • Date of Birth is required (生日是必填的)
      • Basic but good enough for MVP (对于最小可行性产品来说基本但足够好)
      • Bells and whistles (高级功能)
        • Textfield (文本字段)
        • Calendar picker (日历选择器)
    • Bronze plating tricks (青铜版技巧)
      • Manual versus automatic - manual for low infrequent tasks (手动与自动 - 对于低频任务使用手动)
      • Hard-coded versus configurable (硬编码与可配置)
      • Assumed versus asked - default versus personalised (假设与询问 - 默认与个性化)
      • Simple versus complex - address look up versus manual entry (简单与复杂 - 地址查找与手动输入)

总结: 通过青铜和金的流程分析，我们可以定义用户故事的不同版本和实现技巧。

  • Skeleton (框架)
    • Build end to end and then fill in the gaps (从头到尾建立，然后填补空白)
      • Register button > empty form with submit button (注册按钮 > 带提交按钮的空表单)
      • Completes registration (完成注册)
      • Stub out the back end code to focus on UI flow (为了专注于UI流程，使用桩代码代替后端代码)

总结: 框架方法是首先构建整体流程，然后逐步完善细节。
