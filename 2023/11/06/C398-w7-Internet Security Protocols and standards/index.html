P2 MIME 和 S/MIME

MIME 和 S/MIME (MIME与S/MIME)

MIME (多用途互联网邮件扩展)

  • 扩展RFC 822规范：MIME是对旧的RFC 822互联网邮件格式的扩展。
    • RFC 822定义了一个简单的邮件头部格式，包括收件人（To）、发件人（From）和主题（Subject）。
    • 假定邮件内容为ASCII文本格式。
  • 提供新的头部字段：MIME定义了多个新的头部字段，用以描述邮件正文的信息。

总结：MIME为传统的邮件格式提供了扩展，允许邮件包含非ASCII文本内容以及多媒体附件，使邮件内容更为丰富多样。

S/MIME (安全/多用途互联网邮件扩展)

  • 安全邮件扩展：S/MIME是对MIME的安全增强。
    • 增强基于RSA Data Security技术。
  • 支持签名和加密：S/MIME提供了对邮件消息进行签名和/或加密的能力。

总结：S/MIME在MIME的基础上增加了安全特性，包括数字签名和加密，以确保邮件内容的安全性和私密性。

扩展说明：S/MIME不仅支持传统的文本和多媒体邮件格式，还提供了数据完整性验证和隐私保护，广泛应用于需要加密通信的场合，如商务和个人敏感数据交换。

P3 S/MIME内容类型

S/MIME内容类型 (S/MIME Content Types)

多部分 (Multipart)

  • 签名 (Signed)
    • smime Parameter：无
    • 描述：一个清晰签名的消息，包含两部分：一部分是消息本身，另一部分是签名。

应用程序 (Application)

  • pkcs7-mime
    • signedData
      • 描述：一个已签名的S/MIME实体。
    • envelopedData
      • 描述：一个已加密的S/MIME实体。
    • degenerate signedData
      • 描述：一个只包含公钥证书的实体。
    • CompressedData
      • 描述：一个已压缩的S/MIME实体。
  • pkcs7-signature
    • signedData
      • 描述：一个多部分/签名消息的签名内容类型。

总结：
S/MIME提供了多种内容类型以增强邮件安全，包括签名、加密、证书的包含以及消息的压缩。这些内容类型通过对电子邮件进行数字签名和加密来保证信息的安全性，确保信息在传输过程中不被篡改且仅能由授权用户解密和阅读。

扩展说明：

  • 签名：确保消息来源的真实性和完整性。
  • 加密：保护消息内容不被未经授权的第三方读取。
  • 证书：提供了一种机制来验证消息发送者的身份。
  • 压缩：减小消息体积，提高传输效率，也可作为安全措施的一部分。

P4 S/MIME 功能

S/MIME 功能 (S/MIME Functions)

封装数据 (Enveloped data)

  • 加密内容及相关密钥 (Encrypted content and associated keys)
    • 内容加密确保了信息的保密性。
    • 相关密钥用于信息的解密过程。

签名数据 (Signed data)

  • 编码消息加签名摘要 (Encoded message + signed digest)
    • 提供消息的完整性验证和发送者身份的认证。
    • 签名摘要是消息的加密哈希，可用于验证内容未被篡改。

明文签名数据 (Clear-signed data)

  • 明文消息加编码签名摘要 (Cleartext message + encoded signed digest)
    • 允许接收者在解密内容之前验证签名。
    • 明文形式的消息便于阅读，而签名摘要提供安全性。

签名和封装数据 (Signed and enveloped data)

  • 嵌套的签名和加密实体 (Nesting of signed and encrypted entities)
    • 结合签名和加密的双重安全措施。
    • 适用于需要高级安全性的场合，如敏感通讯。

总结：
S/MIME提供多种功能来加强邮件的安全性，包括内容的加密、消息的签名、以及这两者的结合使用。通过使用加密和签名，S/MIME保障了邮件内容的机密性、完整性和认证性，适用于需要高度保密和安全验证的通信。

扩展说明：

  • 加密用于防止未授权访问邮件内容。
  • 签名用于验证消息未被篡改，确保了发送者的真实性。
  • 结合使用提供了更全面的安全策略，适合保护高价值的信息。

P5 S/MIME 流程

S/MIME 功能流程简化版 (Simplified S/MIME Functional Flow)

发送者签名后加密消息 (Sender signs, then encrypts message)

  • 发送者的私钥 (Sender’s private key)
    • 用于对消息进行签名，保证消息的来源和完整性。
  • 签名过程 (Sign)
    • 使用如 RSA/SHA-256 的算法对消息进行签名。
  • 一次性密钥 (One-time secret key)
    • 用于对消息进行加密。
  • 加密过程 (Encrypt)
    • 使用如 AES-128/CBC 的算法对消息及其签名进行加密。
  • 接收者的公钥 (Receiver’s public key)
    • 用于加密一次性密钥，确保只有特定的接收者能解密。

接收者解密消息后验证发送者的签名 (Receiver decrypts message, then verifies sender’s signature)

  • 接收者的私钥 (Receiver’s private key)
    • 用于解密一次性密钥。
  • 发送者生成的密钥 (Secret key generated by sender)
    • 解密消息和签名。
  • 解密过程 (Decrypt)
    • 使用相应的算法如 AES-128/CBC 对消息进行解密。
  • 发送者的公钥 (Sender’s public key)
    • 用于验证消息签名的有效性。

总结：
S/MIME提供了一种机制，通过发送者的私钥签名，再利用接收者的公钥加密的方式，确保了消息传输的安全。在接收端，接收者利用私钥解密，再使用发送者的公钥验证签名，确保了消息的安全性和完整性。

扩展说明：

  • 通过这种方法，S/MIME确保了电子邮件通信的机密性和可验证性。
  • 签名确认了发送者的身份并防止消息在传输过程中被篡改。
  • 加密保护了消息内容不被未授权的第三方读取。
  • 此流程对电子邮件安全尤为重要，特别是在包含敏感信息的商务通讯中。

S/MIME功能流程简化图 (Simplified S/MIME Functional Flow)

概述 (Overview)

S/MIME（安全/多用途互联网邮件扩展）是一个用于MIME数据的公钥加密和签名的标准。

消息 (Message - “Msg”)

  • “Msg” 代表着发送者想要安全传输的原始信息。

签名 (Signature - “Sig”)

  • 发送者使用如SHA-256这样的密码散列函数通过私钥签署消息，生成”Sig”，确保消息的完整性和真实性。
  • 签名允许接收者验证消息未被篡改且确实来自发送者。

流程 (The Process)

  • 签名然后加密 (Sign then Encrypt)
    • (a) 部分显示，发送者首先对消息进行签名，然后使用一次性秘钥对消息和签名进行加密。
    • 一次性秘钥本身使用接收者的公钥加密。
  • 解密并验证 (Decrypt and Verify)
    • (b) 部分显示，接收者使用自己的私钥解密一次性秘钥，进而用它解密消息和签名。
    • 最后，接收者使用发送者的公钥来验证签名。
  • 如果签名验证有效，就确认消息确实由发送者发送且未被更改。

中文总结：
本图描述了S/MIME的功能流程，包括消息的签名和加密过程。发送者通过私钥对消息签名来保证消息的安全性，然后使用接收者的公钥对一次性秘钥进行加密，确保只有特定接收者能解密。接收者通过自己的私钥解密秘钥，并用它解密消息和签名，最后通过发送者的公钥验证签名的有效性。这一流程保证了信息的安全传输和验证。

P6 签名及明文签名

签名及明文签名数据 (Signed and Clear-Signed Data)

用于签名S/MIME消息的首选算法 (The preferred algorithms for signing S/MIME messages)

  • 通常使用RSA或DSA对SHA-256消息哈希进行签名。

签名过程 (The process works as follows)

  • 消息哈希 (Message Hashing)
    • 对要发送的消息使用SHA-256算法，映射为一个固定长度的256位代码。
  • 消息摘要的独特性 (Uniqueness of the Message Digest)
    • 256位的消息摘要对于该消息是唯一的，使得篡改或替换消息而不改变摘要几乎是不可能的。
  • 加密摘要 (Encrypting the Digest)
    • S/MIME使用RSA算法和发送者的私钥对摘要进行加密。
  • 生成数字签名 (Digital Signature Creation)
    • 加密后的摘要即为数字签名，随消息一同发送。
  • 签名的验证 (Signature Verification)
    • 接收方可以使用RSA算法和发送者的公钥对签名进行解密，重新计算消息摘要来验证。
  • 操作效率 (Efficiency of the Operation)
    • 这个过程只涉及对256位块的加密和解密，因而耗时很少。

中文总结：
S/MIME签名过程通过对消息内容进行哈希处理，并使用发送者的私钥加密哈希值，形成独特的数字签名。这个过程保证了消息的完整性和发送者的身份验证，同时因为只处理固定长度的数据，其操作效率很高。

P7 封装数据

Enveloped Data 封装数据

默认加密算法 (Default Algorithms)

  • S/MIME消息加密通常使用的算法有AES和RSA。
    • S/MIME生成一个伪随机的秘钥用于通过AES或其他传统加密方案加密消息。
    • 每条新消息都会生成一个新的伪随机密钥。
    • 会话密钥与消息绑定并一同传输。

加密过程 (Encryption Process)

  • 秘钥作为公钥加密算法RSA的输入，用接收者的公钥来加密。
  • 在接收端，S/MIME利用接收者的私钥解密以获取秘钥，然后用秘钥和AES恢复明文消息。

数据转换 (Data Conversion)

  • 如果只使用加密，使用radix-64将密文转换为ASCII格式。

中文总结：
本段介绍了S/MIME在封装数据时使用的默认加密算法，以及加密和解密的整个过程。首先，为每条消息生成一个伪随机的秘钥，然后用接收者的公钥通过RSA算法加密该秘钥。当消息被接收时，接收者使用自己的私钥解密这个秘钥，再使用AES算法解密消息内容，获取原始信息。如果需要，还可以将密文转换为ASCII格式以便传输。这个过程确保了传输信息的安全性和私密性。

加密算法简介

AES (高级加密标准)

  • 类型: 对称加密算法。
  • 特点:
    • 使用相同的密钥进行加密和解密。
    • 美国联邦政府采用的加密标准。
    • 适合快速加密大量数据。
  • 用途: 广泛用于商业和政府的数据保护。

RSA (Rivest-Shamir-Adleman)

  • 类型: 非对称加密算法。
  • 特点:
    • 使用一对密钥，公钥加密，私钥解密。
    • 安全性依赖于大数分解问题。
    • 加密速度慢，但非常安全。
  • 用途:
    • 安全通信，如SSL/TLS协议。
    • 数字签名验证。

中文总结：
AES和RSA是两种常用的加密算法，AES是对称加密，适合加密大量数据，而RSA是非对称加密，主要用于安全通信和数字签名。在实际使用中，通常会结合这两种算法，使用RSA加密AES的密钥，然后再用AES加密数据，以此来保证数据传输的安全性。

扩展解释：

  • 对于AES，目前最常用的密钥长度有128位、192位和256位，其中256位的AES加密被认为是非常安全的。
  • RSA在加密大型数据时可能不够高效，因此它更多的是用来加密小数据量，例如用于加密AES密钥这样的敏感信息。

P8 DKIM

DomainKeys Identified Mail (DKIM)

DKIM 规范

  • 加密签名: DKIM 允许使用加密签名来验证电子邮件的发送域。
  • 域名认证: 签名域名对电子邮件中的消息负责。

DKIM 标准

  • 互联网标准草案: DKIM 已被提议为互联网标准，具体在 RFC 4871: DomainKeys Identified Mail (DKIM) Signatures 中定义。

DKIM 的普及

  • 广泛应用: 多个电子邮件提供商已经普遍采用 DKIM。

中文总结：
DKIM 是一种电子邮件验证方法，它通过加密签名来验证电子邮件是否来自声明的发送域。这有助于减少垃圾邮件和网络钓鱼攻击，因为接收者可以确认邮件没有被篡改，并且确实是由声明的发送者所发送。RFC 4871 标准定义了如何在电子邮件中实施 DKIM 签名。

扩展解释：

  • DKIM 使用公钥加密技术，发送方的邮件服务器将包含 DKIM 签名的邮件发送出去，接收方的邮件服务器利用发送方的公钥验证签名。
  • 如果邮件经过未经授权的更改，签名将不会匹配，接收方的邮件服务器可以据此判断邮件可能是伪造的。
  • DKIM 使电子邮件更加安全，提高了电子邮件系统的整体信任度。

P9 互联网邮件架构

Internet Mail Architecture（互联网邮件架构）

概述

  • 互联网邮件架构由多个功能模块和标准化协议组成，它们定义了邮件在不同模块间的传输方式。

功能模块

  • Message User Agent (MUA): 用户代理，用于创建和读取邮件。
  • Mail Submission Agent (MSA): 邮件提交代理，用于将邮件从MUA提交到邮件处理系统。
  • Message Transfer Agent (MTA): 邮件传输代理，负责在邮件处理系统中传输邮件。
  • Mail Delivery Agent (MDA): 邮件投递代理，用于将邮件从MTA传递到收件人的邮件存储。
  • Message Store (MS): 邮件存储，用于存储接收到的邮件。

标准化协议

  • Simple Mail Transfer Protocol (SMTP): 用于MTA之间邮件传输的协议。
  • Extended SMTP (ESMTP): SMTP的扩展，用于邮件提交时提供额外的命令和功能。
  • Internet Message Access Protocol (IMAP) 和 Post Office Protocol (POP): 用于MUA从MS获取邮件的协议。

中文总结：
互联网邮件架构涉及邮件的创建、提交、传输、存储和检索。用户代理（MUA）创建邮件，通过邮件提交代理（MSA）和邮件传输代理（MTA）在邮件处理系统内传输，最终由邮件投递代理（MDA）投递到邮件存储（MS），收件人通过MUA检索邮件。这个过程涉及多个协议，包括SMTP、ESMTP、IMAP和POP。

扩展解释：

  • 这些组件和协议共同工作，确保邮件能够可靠地从发送者发送到接收者。
  • SMTP用于发送邮件时的服务器到服务器通信，而IMAP和POP用于收件人从他们的邮件服务器上检索邮件。
  • 安全性可以通过如TLS这样的协议来增强，以保护邮件在传输过程中的安全。
  • DKIM和SPF等技术可以帮助验证邮件的发送源，防止垃圾邮件和网络钓鱼攻击。

P10 DKIM部署示例

DKIM Deployment Example（DKIM部署示例）

邮件发送网络

  • MUA (Message User Agent): 用户的电子邮件客户端，用于撰写和发送邮件。
  • MSA (Mail Submission Agent): 接收来自MUA的邮件并转发到MTA。
  • MTA (Message Transfer Agent): 负责在邮件服务器间转发邮件。
  • DNS (Domain Name System): 存储公钥和其他配置，以供验证邮件。

邮件接收网络

  • MDA (Mail Delivery Agent): 从MTA接收邮件并投递到收件人的邮箱。
  • MUA (Message User Agent): 收件人的电子邮件客户端，通过POP或IMAP协议接收邮件。

DKIM工作流程

  • 发送邮件时，MSA会用发送域的私钥对邮件进行签名。
  • 签名的邮件通过MTA发送，并在传输过程中可被SMTP协议处理。
  • 接收方的MTA通过DNS查询发送方域的公钥，对签名进行验证。
  • 验证成功后，MDA将邮件投递到收件人的邮箱中。

中文总结：
图表展示了使用DKIM技术进行电子邮件发送和接收的简化流程。DKIM通过在DNS中存储公钥来验证邮件来源，确保邮件的真实性和完整性。发送方使用私钥签名邮件，接收方通过DNS查询公钥验证签名。邮件系统通过SMTP、POP、IMAP等协议处理邮件的传输和接收。

扩展解释：

  • DKIM是防止电子邮件伪造的重要技术，通过加密签名确保邮件内容和头部信息没有被篡改。
  • DKIM允许邮件接收者验证发送者的域名，增强了邮件系统的安全性。
  • 这个过程不需要邮件用户的参与，完全在邮件系统后台自动完成。
  • 尽管DKIM能够提供邮件发送方的验证，但它并不加密邮件内容，这通常需要结合其他技术，如S/MIME，来实现。

P11 安全套接层和传输层安全性

Secure Sockets Layer (SSL) and Transport Layer Security (TLS)（安全套接层和传输层安全性）

SSL/TLS 概述

  • 广泛应用的安全服务：SSL和TLS是最广泛使用的安全服务之一。
  • 通用服务：它们是作为依赖于TCP的一套协议实现的，提供通用的安全服务。
  • 成为互联网标准：TLS后来成为了互联网标准RFC 4346。

实现选择

  • 作为基础协议套件的一部分提供：SSL/TLS可以作为底层协议套件的一部分提供，例如作为网络应用或操作系统的一部分。
  • 嵌入到特定的软件包中：SSL/TLS也可以被嵌入到特定的软件包中，以便于为特定的应用提供加密支持。

中文总结：
SSL和TLS是实现网络通信安全的关键技术。SSL是早期的安全协议，而TLS是其后续版本，成为了新的标准。这两种协议都提供了数据加密、服务器认证、消息完整性检验等安全服务，保护数据在互联网中的安全传输。它们可以作为整体网络协议的一部分提供，也可以作为特定应用的一部分嵌入实现。

扩展解释：

  • SSL/TLS通过加密传输的数据来防止数据被截获或篡改。
  • 它们使用公钥和私钥来实现安全的密钥交换。
  • RFC 4346是TLS的一个标准化文档，它定义了TLS协议的具体实现和要求。
  • 在实际应用中，SSL/TLS通常被用于安全的网页浏览（HTTPS）、电子邮件传输安全（SMTPS、IMAPS、POPS）和其他需要加密的网络通信中。

P12 SSL/TLS 协议栈

SSL/TLS Protocol Stack（SSL/TLS 协议栈）

SSL/TLS 协议层次结构

  • **握手协议 (Handshake Protocol)**：负责在通信双方之间建立加密通信的参数。
  • **更改密码规范协议 (Change Cipher Spec Protocol)**：用于在握手协议完成后切换到新协商的密码规范。
  • **警报协议 (Alert Protocol)**：用于在出现错误时通知对方。
  • HTTP：表明SSL/TLS可以应用于HTTP协议，从而形成HTTPS。
  • **心跳协议 (Heartbeat Protocol)**：用于保持连接活跃，同时检测另一端是否仍然响应。

底层协议

  • **记录协议 (Record Protocol)**：为上层协议提供基础的加密传输服务，并确保数据的完整性。
  • TCP：传输控制协议，是一种可靠的、面向连接的、基于字节流的传输层通信协议。
  • IP：互联网协议，负责路由数据包从发送者到接收者。

中文总结：
SSL/TLS 协议栈是在TCP/IP模型的基础上，为网络通信提供安全保障的一系列协议。它包括握手协议以建立加密通信参数，更改密码规范协议以启用新的加密方式，警报协议以通知错误情况，以及心跳协议以维持连接。这些都是建立在记录协议之上的，后者为所有上层协议提供安全和数据完整性服务。整个协议栈运行在标准的TCP/IP网络模型之上。

扩展说明：

  • 握手协议是SSL/TLS协议栈中最复杂的部分，包括密钥交换、服务器认证、客户端认证和协商加密算法。
  • 记录协议定义了SSL/TLS数据的封装格式，保证数据私密性和可靠性。
  • SSL/TLS通常被用于Web浏览器和服务器之间的加密通信，即我们熟知的HTTPS。
  • 心跳协议在保持长连接和网络设备的实时状态检测中发挥着重要作用。

P13 TLS 概念

TLS Concepts（TLS 概念）

TLS 会话 (TLS Session)

  • 客户端与服务器之间的关联：TLS会话是客户端和服务器之间的一种关联，它们共享安全设置。
  • 通过握手协议创建：握手协议用于在会话初始化时建立这种关联。
  • 定义一组加密安全参数：这些参数包括加密算法、密钥、证书等信息。
  • 用于避免每个连接的昂贵安全参数协商：一旦会话建立，相同的安全参数将被用于后续的连接，节省了重复协商的成本。

TLS 连接 (TLS Connection)

  • 提供适当服务类型的传输：在OSI模型中，TLS连接提供端到端的安全传输服务。
  • 点对点关系：TLS连接是建立在两个端点之间的，保证了数据从一端传输到另一端的安全性。
  • 瞬时的：TLS连接是临时的，通常只在数据传输期间存在。
  • 每个连接都与一个会话相关联：这意味着连接可以复用会话期间协商的参数。

中文总结：
TLS会话和TLS连接是传输层安全协议中的两个核心概念。TLS会话为客户端和服务器之间的多个连接建立了一个长期的关联，其中安全参数只需协商一次。而TLS连接是基于这些参数建立的瞬时安全通道，用于具体的数据传输。这样的设计有效提高了通信的效率和安全性。

扩展说明：

  • TLS会话的建立依赖于TLS握手协议，该协议确保了通信双方的身份验证以及加密密钥的协商。
  • 一旦TLS会话建立，它可以被多个TLS连接复用，从而避免了每次连接都进行复杂的密钥交换和协议协商。
  • TLS连接确保了每次数据传输的机密性和数据完整性，即使在开放和不安全的网络中也能保证数据的安全。

P14 TLS 记录协议操作

TLS Record Protocol Operation（TLS 记录协议操作）

TLS协议处理应用数据的过程包含以下步骤：

应用数据分段 (Application Data Fragmentation)

  • 数据分段：将大块的应用数据分割成更小的片段，以便于处理。

数据压缩 (Data Compression)

  • 压缩数据：对分割后的数据进行压缩，减少数据量，提高传输效率。

MAC 添加 (MAC Addition)

  • 添加消息认证码（MAC）：对每个数据片段计算MAC，确保数据完整性和认证信息的来源。

数据加密 (Data Encryption)

  • 加密数据：使用会话密钥对数据和MAC进行加密，保障数据在传输过程中的机密性。

SSL 记录头添加 (SSL Record Header Appending)

  • 添加SSL记录头：在加密后的数据前附加一个SSL记录头，包含了协议版本和加密数据的长度等信息。

中文总结：
TLS记录协议通过对数据进行分段、压缩、加密和添加记录头等一系列步骤，确保了数据在传输过程中的安全性。这个过程涉及了数据的完整性校验、机密性保护以及协议信息的标识，是确保TLS能够安全传输数据的关键部分。

扩展说明：

  • 数据分段的目的是为了匹配底层网络协议如TCP/IP的数据处理能力，同时也便于在发生错误时只重传小片段的数据，而不是整个大块数据。
  • 数据压缩可以减少传输的数据量，但要注意压缩过程中可能存在的安全漏洞，例如CRIME和BREACH攻击。
  • MAC不仅保证了数据的完整性，也防止了重放攻击，因为它包含了一个序列号。
  • 数据加密确保了数据内容的保密，防止中间人攻击，但前提是使用了强加密算法和安全的密钥交换机制。
  • SSL记录头则为数据解密和处理提供了必要的上下文信息。

P15 变更加密规范协议

Change Cipher Spec Protocol（变更加密规范协议）

TLS协议中的变更加密规范协议是一个简单的协议，用于更新加密套件和安全参数：

  • TLS协议组成：变更加密规范协议是TLS协议四个特定协议之一，用于TLS记录协议。
  • 协议简易性：它是所有TLS协议中最简单的。
  • 消息结构：该协议只包含一个单字节消息，其值为1。
  • 消息目的：这个消息的唯一目的是触发待定状态的变更，并将其复制到当前状态。
  • 结果：因此，这个操作更新了正在使用的加密套件。

中文总结：
变更加密规范协议是TLS安全协议中的一个重要组成部分，它通过发送一个简单的信号来更新通信双方的加密参数。这是保证TLS会话中加密套件能够及时更新，以维护通信安全的一个关键步骤。

扩展说明：

  • 这个过程在TLS握手过程中非常重要，尤其是在客户端和服务器协商加密算法和密钥之后，确保双方使用最新协商的安全参数进行加密通信。
  • 此消息的简单性确保了TLS协议的高效性，能夜快速地在保持安全的同时更新会话状态。

P16 警报协议

Alert Protocol（警报协议）

TLS协议中的警报协议用于传递安全相关的警报信息：

  • 警报传递：向对等实体传递与TLS相关的警报。
  • 消息加密：警报消息经过压缩和加密处理。
  • 消息结构：
    • 第一个字节：取值为警告（1）或严重（2），表明消息的严重程度。
    • 第二个字节：包含指示特定警报类型的代码。
  • 错误处理：
    • 如果是致命级别，TLS会立即终止连接。
    • 对于非致命警报，当前会话的其他连接可能会继续，但不允许建立新的连接。

中文总结：
警报协议是TLS中用于通知和处理安全问题的一种机制。当TLS检测到潜在的安全问题时，它会使用警报协议向对方发送消息。如果问题严重到足以影响通信安全，那么TLS会立刻中断该连接，以避免数据泄露或其他安全问题。

扩展说明：

  • 警报协议的设计考虑到了不同安全问题的严重程度，允许在不同情况下采取不同的措施，从而提供了灵活的错误处理方式。
  • 对警报消息的加密保证了即使在公共网络中传输也不会被轻易截获，增强了警报机制的安全性。

P17 握手协议

Handshake Protocol（握手协议）

握手协议是TLS（传输层安全性协议）中最为复杂的部分：

  • 前置条件：在任何应用数据被传输之前使用。
  • 功能：允许服务器和客户端进行以下操作：
    • 相互认证：验证对方的身份。
    • 协商加密和MAC算法：确定用于加密和消息认证的算法。
    • 协商将使用的加密密钥：确定通信双方用于加密的密钥。
  • 交换过程：由客户端和服务器交换的一系列消息组成。
  • 交换阶段：握手过程包含四个阶段。

中文总结：
握手协议是确保TLS通信安全的关键步骤，它在数据传输前验证通信双方的身份，并确定加密参数和密钥。这一过程确保了数据传输的安全性和完整性。

扩展说明：

  • 握手协议的设计使得只有在确认双方的身份并同意安全参数之后，才能开始实际的数据传输，从而大大提升了通信过程的安全性。
  • 握手过程中使用的复杂性和多个阶段，虽然增加了建立连接的时间，但这是为了确保建立的连接具有高度的安全保障。

P18 握手协议行为

Handshake Protocol Action（握手协议行为）

图表展示了TLS握手协议的具体动作和时间线，包括客户端和服务器之间的交互过程：

  • Phase 1（阶段1）

    • client_hello：客户端向服务器发送问候，建立安全能力，包括协议版本、会话ID、密码套件、压缩方法和初始随机数。
    • server_hello：服务器响应客户端问候，确认安全设置。

  • Phase 2（阶段2）

    • certificate：服务器发送其证书。
    • server_key_exchange（可选）：服务器发送密钥交换信息。
    • certificate_request（可选）：服务器请求客户端的证书。
    • server_hello_done：服务器完成问候消息阶段。

  • Phase 3（阶段3）

    • certificate（可选）：客户端发送其证书（如果被请求）。
    • client_key_exchange：客户端发送密钥交换信息。
    • certificate_verify（可选）：客户端发送证书验证。

  • Phase 4（阶段4）

    • change_cipher_spec：客户端和服务器通知对方将开始使用新的加密参数。
    • finished：客户端和服务器发送完成消息，握手协议结束。

中文总结：
TLS握手协议是一系列在客户端和服务器之间交换的消息，目的是为了在数据传输前建立一个安全的通信通道。每个阶段都是建立TLS安全连接的必要步骤，以确保数据传输的安全性。

扩展说明：

  • 阶段1和2主要关注于安全参数的建立和身份验证。
  • 阶段3是密钥信息的交换，确保了后续通信的加密。
  • 阶段4是双方确认将使用新的加密参数，并正式开始加密通信。

P19 心跳协议

Heartbeat Protocol（心跳协议）

心跳协议是用于维持和监控网络连接状态的协议：

  • 产生周期性信号，由硬件或软件生成，指示系统正常运行或同步系统其他部分。
  • 常用于监测协议实体的可用性。
  • 在2012年通过RFC 6250标准定义。
  • 基于TLS记录协议之上运行。
  • 在握手协议的第一阶段建立使用。
  • 每个节点表明它是否支持心跳。
  • 服务于两个目的：
    • 确保发送方，接收方仍然在线。
    • 在闲置期间产生连接活动。

中文总结：
心跳协议是TLS连接中用于保持连接活性和状态检测的关键部分，能够确保即便在没有数据传输的时候连接也不会因超时而关闭。此外，它还帮助同步分布式系统的不同部分，保证整个系统的协调运行。

扩展说明：
心跳协议的设计允许在网络连接期间定期检查对端是否仍然响应，这在长时间运行的连接中尤为重要。通过心跳消息，一旦检测到连接问题，可以迅速采取措施，如重连或报错，提高系统的稳定性和可靠性。

P20 SSL/TLS 攻击

SSL/TLS Attacks（SSL/TLS 攻击）

SSL/TLS攻击通常可以分为四个一般类别：

  • Attacks on the Handshake Protocol（握手协议攻击）
    • 这类攻击目标是TLS握手过程，目的是破坏密钥交换或认证过程。
  • Attacks on the record and application data protocols（记录和应用数据协议攻击）
    • 此类别的攻击尝试破坏或篡改传输中的数据，如通过记录层协议进行中间人攻击。
  • Attacks on the PKI (Public Key Infrastructure)（公钥基础设施攻击）
    • 这些攻击针对的是证书颁发体系，比如伪造或劫持证书的攻击。
  • Other attacks（其他攻击）
    • 包括针对SSL/TLS协议实现的各种其他攻击，比如著名的心脏出血（Heartbleed）漏洞。

中文总结：
SSL/TLS攻击是针对网络通信安全协议的攻击，它们试图通过不同的方式破坏安全通信，攻击的手段多种多样，包括但不限于破坏握手过程、数据传输、证书系统以及利用协议实现中的漏洞。

扩展说明：
随着网络安全威胁的不断演变，攻击者可能会发现新的攻击方法来针对SSL/TLS协议。因此，了解这些攻击的类型和工作方式对于维护网络安全至关重要，同时也需要不断更新和加固协议的实现来防范新出现的威胁。

P21 心脏出血漏洞

The Heartbleed Exploit（心脏出血漏洞）

心脏出血漏洞是一个严重的安全漏洞，主要影响使用TLS心跳协议的服务器。以下是心脏出血漏洞的工作原理及其对系统的影响：

(a) How TLS Heartbeat Protocol works（TLS心跳协议工作原理）

  • CLIENT 发送心跳请求消息（HEARTBEAT REQUEST MESSAGE）。
    • 确保接收到的有效负载与发送的相同。
  • SERVER 提取负载并将其放入响应消息中（HEARTBEAT RESPONSE MESSAGE）。

(b) How TLS Heartbleed exploit works（TLS心脏出血漏洞工作原理）

  • CLIENT 发送畸形的心跳请求：小负载伪装成大负载。
  • SERVER 从内存数据中提取负载并放入响应消息。
    • 由于预期负载会很大，因此“桶”也会获取其他数据。
  • 收到的心跳响应中可能包含敏感信息：
    • TLS私钥
    • 认证cookie
    • 密码/凭证

中文总结：
心脏出血漏洞利用了TLS心跳协议的一个实现缺陷，攻击者可以通过发送错误构造的心跳请求来导致服务器泄露内存中的敏感数据。这一漏洞的发现凸显了即使是广泛使用的安全协议也可能存在严重的安全风险。

扩展说明：
这个图表清晰地展示了心脏出血漏洞的攻击过程，突出了安全实践中的一个重要教训：安全协议的实现需要经过严格的审查和测试，以防止此类潜在的安全威胁。心脏出血漏洞的发现促使整个安全社区对现有安全措施进行了重新评估，以确保数据传输的安全性。

P22 超文本传输安全协议

HTTPS (HTTP over SSL)（HTTPS: 超文本传输安全协议）

HTTPS 是一种安全通信协议，结合了 HTTP 和 SSL/TLS 协议来在 Web 浏览器和 Web 服务器之间提供加密通信。

  • 组合：HTTP 和 SSL 的结合，用于实现 Web 浏览器和 Web 服务器之间的安全通信。
  • 普及性：集成在所有现代 Web 浏览器中。
    • 搜索引擎可能不支持 HTTPS。
    • URL 地址以 https:// 开头。
  • 文档化：在 RFC 2818 “HTTP Over TLS” 中有所描述。
  • 客户端代理：作为 HTTP 客户端的代理同时充当 TLS 客户端。
  • 连接关闭：关闭一个 HTTPS 连接需要关闭与远端 TLS 实体的连接，这通常涉及关闭底层的 TCP 连接。

中文总结：
HTTPS 是现代互联网中用于保护用户数据安全、隐私和完整性的关键技术。它通过结合 HTTP 和 SSL/TLS 协议，确保了用户数据在传输过程中的加密和安全性。

扩展说明：
在当前的网络环境中，HTTPS 已经成为网站标准配置，特别是对于处理敏感数据的网站，如电子商务、在线银行等，HTTPS 提供了一种必要的保护措施，以防止数据在传输过程中被拦截或篡改。

P23 IP 安全

IP Security (IPsec)（IP 安全）

IPsec 是一种网络协议套件，用于在 IP 层上保护网络通信，确保数据包的完整性、认证以及加密。

  • 应用层安全机制：包括 S/MIME、Kerberos、SSL/HTTPS 等不同的应用层安全机制。
  • 跨协议层的安全问题：IPsec 旨在处理在各个协议层上出现的安全问题。
  • 全网络安全实现：希望通过网络为所有应用提供安全性。
  • 认证和加密功能：在新一代的 IPv6 中内置了认证和加密安全功能。
  • IPv4 的兼容性：也可以在现有的 IPv4 网络中使用。

中文总结：
IPsec 提供了在传输层实施安全措施的方法，它支持多种安全机制并且可以跨多个协议层工作。这使得 IPsec 成为保护通信安全的重要工具，特别是在新的 IP 地址版本 IPv6 中，它已成为标准配置。

扩展说明：
随着网络安全威胁的不断演变，IPsec 的重要性不断增加。它通过在 IP 层提供加密和认证服务，为数据传输提供端到端的安全性。此外，IPsec 的设计允许它灵活地应对各种安全需求，并且能够与老旧系统兼容，使其成为当前和未来网络安全架构的关键组成部分。

P24 IPsec 的好处

IPsec 的好处（Benefits of IPsec）

IPsec 提供了多种好处，从强化网络安全到为用户和设备提供透明的保护。

  • 加强边界安全：当在防火墙或路由器中实施时，IPsec 为所有跨越边界的流量提供强有力的安全保障。
  • 防火墙抗绕过：在防火墙中，IPsec 能够抵御绕过尝试，增强安全性。
  • 透明性：因为它位于传输层以下，对应用程序来说是透明的。
  • 对终端用户透明：IPsec 的使用对终端用户可以是完全透明的，不需要用户干预。
  • 为个人用户提供安全：IPsec 可以为个别用户提供安全服务。
  • 保护路由架构：IPsec 保障了路由架构的安全，防止路由攻击。

中文总结：
IPsec 是一种强大的网络安全工具，它不仅为数据交换提供了坚固的安全屏障，而且其操作对用户来说是透明的，这使得它成为个人和企业首选的安全解决方案。

扩展说明：
IPsec 特别适用于需要跨越不安全网络传输敏感数据的场景，如跨公共互联网的企业通信。它的透明性和安全性使得用户在享受网络服务的同时，无需担忧其数据的安全性。此外，IPsec 的适用性跨越了多种网络设备，使其成为保护网络边界的有力工具。

P25 IPsec 的范围

IPsec 的范围（The Scope of IPsec）

IPsec 提供了两个主要功能，支持虚拟私人网络（VPN）的需要，并且其规范相当复杂。

  • 主要功能：

    • 组合认证/加密功能：被称为封装安全有效载荷（ESP）的组合认证和加密功能。
    • 密钥交换功能：确保通信双方能够安全地交换密钥信息。

  • 认证功能：

    • 仅认证功能：使用认证头（AH）实现的认证功能。
    • 向后兼容性：虽然消息认证由ESP提供，但AH仍包含在IPsec的第三版（IPsecv3）中，用于向后兼容，但不推荐在新应用中使用。

  • 复杂的规范：

    • 多个 RFC 文档：IPsec 的规范涵盖了多个RFC文档，如RFC 2401、4302、4303 和 4306，展示了其规范的复杂性。

中文总结：
IPsec 为VPN提供了认证和加密的综合解决方案。它通过ESP提供了一种既能认证也能加密的功能，同时还有专门的密钥交换机制。尽管存在仅用于认证的AH功能，但随着技术的发展，现在主要推荐使用ESP。此外，IPsec 的规范非常复杂，涉及多个RFC文档，这也反映了其在网络安全领域的深度和广度。

扩展说明：
IPsec是网络安全中的关键技术，它不仅在VPN构建中起到核心作用，而且其应用已经扩展到多种网络通信环境。ESP和AH的设计确保了传输数据的完整性、认证性和机密性。虽然规范复杂，但IPsec的实施为网络通信提供了坚实的安全基础。

P26 安全关联

安全关联（Security Associations）

安全关联（SA）是 IPsec 中定义通信双方之间一种重要的机制。

  • 一般描述：

    • 单向关系：发送者和接收者之间的单向关系，用于保障流量流转的安全。
    • 双向通信：如果需要双向安全通信，则需要两个安全关联。

  • 唯一标识：

    • 目的地址：通过 IPv4 或 IPv6 头部中的目的地址唯一标识。
    • SPI：在封装的扩展头（AH 或 ESP）中的安全参数索引（SPI）。

  • 定义参数：

    • 安全参数索引（SPI）：用于在 IPsec 协议中标识特定的安全关联。
    • IP 目的地址：数据包发送到的地址。
    • 协议标识符：表明使用的是 AH 或 ESP 协议。

中文总结：
安全关联是保护 IPsec 通信双方流量安全的基础结构，它通过一系列参数确保数据传输的安全性和可追溯性。每个安全关联由目的地址和 SPI 唯一确定，并用于具体的安全协议（如 AH 或 ESP）中，以保障数据的完整性和机密性。

扩展说明：
在一个完整的 IPsec 实现中，安全关联作为构成安全策略的基石，对于确保数据包的加密和解密过程至关重要。它们的存在使得数据包能够在发送和接收时保持加密状态，并确保只有经过验证的接收者才能解密和读取内容。

图表解释：

P27 IPsec ESP 格式

IPsec ESP 格式（IPSec ESP Format）

IPsec 封装安全有效载荷（ESP）提供了数据机密性、数据源认证和消息完整性。

  • 安全参数索引（SPI）：

    • 位于头部的前 32 位，用于连接到相关的安全协会。

  • 序列号（Sequence Number）：

    • 保护 against 重放攻击，通过序列化加密数据包。

  • 有效载荷数据（Payload Data）：

    • 包含实际传输的信息，其长度可变。

  • 填充（Padding）：

    • 由 0 到 255 字节不等，用于满足加密算法的块大小要求。

  • 填充长度（Pad Length）：

    • 指明填充的长度，以便接收方可以正确地处理信息。

  • 下一个头部（Next Header）：

    • 指示在处理完 ESP 头部后应该处理的下一个协议。

  • 认证数据（Authentication Data）：

    • 提供数据完整性和数据源认证。

中文总结：
图表展示了 IPsec 的 ESP 格式，说明了每个部分的功能和它们在数据安全中的作用。SPI 和序列号提供连接和防重放功能，有效载荷数据承载实际信息，填充和填充长度支持加密操作，而认证数据确保数据的完整性和来源的可靠性。

扩展说明：
ESP 是 IPsec 协议中重要的一部分，它不仅保护数据免受窃听，还验证数据的完整性和发送者的身份，确保网络通信安全。这个格式的详细解释有助于网络安全专业人员理解并实施 IPsec 协议。

传输模式与隧道模式（Transport and Tunnel Modes）

IPsec 有两种模式：传输模式和隧道模式，它们在保护 IP 数据包方面有不同的应用和作用。

传输模式（Transport Mode）

  • 作用范围：只扩展到 IP 数据包的有效载荷部分。
  • 通常用途：用于两个主机之间的端到端通信。
  • 加密与认证：ESP 可以对 IP 有效载荷进行加密，并可选地对其进行认证，但不包括 IP 头部。

隧道模式（Tunnel Mode）

  • 保护范围：对整个 IP 数据包提供保护。
  • 数据传输方式：整个原始数据包通过隧道从 IP 网络的一个点传输到另一个点。
  • 使用场景：当安全协会的一端或双端是安全网关时使用。
  • 网络应用：即使没有实施 IPsec，网络中的许多主机也可以通过防火墙进行安全通信。

中文总结：
传输模式和隧道模式是 IPsec 中用于保护数据传输的两种主要方式。传输模式主要用于保护两个主机之间的通信有效载荷，而隧道模式则提供了更全面的保护，适用于通过安全网关或隧道进行通信的情况。

扩展说明：
这两种模式各自适应不同的安全需求和网络架构。在选择合适的模式时，需要考虑数据保护的范围、通信的起点和终点、以及是否需要通过中间设备（如防火墙或网关）进行路由。
