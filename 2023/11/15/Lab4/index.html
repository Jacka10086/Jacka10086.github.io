实验练习 - IPv4 (Lab Exercise - IPv4)

目标 (Objective)

  • 学习IP（互联网协议）的细节。IP是整个互联网中使用的网络层协议。本练习将专注于IPv4，因为它被广泛部署，而IPv6的使用则较为有限。

方法 (Procedure)

  • 使用traceroute（在Linux和Mac上）或tracert（在Windows上）命令来发现从你的计算机到远程互联网主机的路由层路径。
  • 这个命令在Windows上预装，在Linux上可以通过包管理器安装。
  • 虽然该命令有多种选项，但简单地执行命令traceroute www.uwa.edu.au会使你的计算机找到并打印到远程计算机的路径。

本练习的总结：通过使用网络工具traceroute或tracert，可以了解数据在网络层中如何从源头到达目的地，这有助于理解IP协议的工作原理以及网络故障排查。

实验步骤1：捕获一个追踪 (Step 1: Capture a Trace)

目的 (Objective)

  • 学习IP（互联网协议）细节，特别是IPv4，这是目前互联网上广泛部署的协议。

方法 (Procedure)

  1. 选择一个远程服务器的URL，如 http://www.uwa.edu.au，并检查是否能够使用 wget 或 curl 命令获取该资源。
  2. 执行 traceroute 到相同的远程服务器，以检查网络路径。在Windows上使用 tracert www.uwa.edu.au，在Linux/Mac上使用 traceroute www.uwa.edu.au。
    • 如果你在Linux/Mac上并且在NAT后面，可以使用 -I 选项来使 traceroute 发送ICMP探测。

注意事项 (Points to Note)

  • 如果 traceroute 无法正确工作，则路径上几乎所有的行都将显示 *。这种情况下，应尝试不同的远程服务器或使用提供的追踪。

步骤2：检查追踪 (Step 2: Inspect the Trace)

  • 选择追踪中的任意数据包，并展开IP头部字段以查看详细信息。IP头部是我们感兴趣的部分，它包含了数据包的结构。

字段解析 (Field Analysis)

  • **版本字段 (Version field)**：设置为4，表明这是IPv4。
  • **头部长度字段 (Header length field)**：通过查看数据包中选定的字节，可以观察到版本和头部长度打包在一个字节内。
  • **区分服务字段 (Differentiated Services field)**：包含位标志，指示是否应在路由器中处理数据包的优先级和拥塞通知。
  • **总长度字段 (Total Length field)**：显示整个IP数据包的长度。
  • **标识字段 (Identification field)**：用于当大型IP数据包被分割成多个较小片段时的分组。
  • **标志和片偏移字段 (Flags and Fragment offset fields)**：表示是否更多片段以及片段的偏移。
  • **生存时间字段 (Time to live or TTL field)**：后跟协议字段 (Protocol field)。
  • **头部校验和 (Header checksum)**：检查是否有错误的校验和。

注意事项 (Points to Note)

  • 对于IP数据包来自你的计算机还是远程服务器，操作系统软件会将头部校验和留空供网卡计算和填充。
  • 如果Wireshark认为校验和是错误的，这可能是因为协议卸载的结果。
  • 校验和字段在TCP/IP通常适用。
  • 在头部字段中，源地址和目的地址字段也很重要。

步骤2总结：本节详细介绍了如何在Wireshark中检查和理解IPv4数据包头部的不同字段，以及如何正确解读这些信息。

步骤3：IP数据包结构 (Step 3: IP Packet Structure)

  • 为了展示对IP协议的理解，我们将讨论如何使用Wireshark查看IP数据包并回答一些相关问题。

IP数据包分析 (IP Packet Analysis)

  • 观察IP数据包的位置和IP头部字段的大小，这些可以在Wireshark中观察到。
  • 注意，当你在中间面板点击协议块时，Wireshark会突出显示数据包底部面板中对应的字节，并在窗口底部显示总的数据包大小。

问题及回答 (Questions and Answers)

  1. 你的计算机和远程服务器的IP地址是什么？
  2. 总长度字段是否包括IP头部加上有效载荷，还是仅仅是IP有效载荷？
  3. 标识字段的值是否在一个TCP连接的所有数据包中保持不变，还是每个包都有所不同？它在两个方向上是否相同？值变化时是否有任何模式？
  4. 从你的计算机发送的数据包中TTL字段的初始值是多少？它是最大可能值，还是更低的值？
  5. 如何从查看数据包判断它没有被分片？
  6. IP头部的长度是多少，这在头部长度字段中如何编码？

步骤3总结：本节介绍了如何通过Wireshark分析IP数据包的结构，并通过回答相关问题来深化对IP数据包构成的理解。

步骤4：互联网路径 (Step 4: Internet Paths)

在这一步骤中，我们使用 traceroute 工具来探索数据包在互联网上从源地址到目的地址之间的路径。

IP地址 (IP Addresses)

  • IP数据包的源地址和目的地址标记了互联网路径的端点。
  • traceroute 通过引发ICMP TTL超时消息从每个路由器中获取响应，以发现从源头到目的地的网络路径。

路径追踪 (Traceroute)

  • 使用 traceroute 输出，绘制网络路径图。
  • 这个工具通常会提供每跳的往返时间（RTT）和路由器的IP地址及DNS名称。
  • traceroute 输出的例子在图4中展示了路由级别的路径。

绘制网络路径 (Sketching the Network Path)

  • 展示计算机（左侧）和远程服务器（右侧）的IP地址，以及它们之间的路由器IP地址。
  • 使用 traceroute 输出，标记路径上的路由器及其归属的实际组织名称。

注意事项 (Points to Note)

  • 对于不同的路由器实现，没有保证 traceroute 总是能工作。
  • 常见的 * 响应表示路径的某些部分失败了。

分析和标记 (Analysis and Labeling)

  • 如果不确定，将路由器与你认为的组织域名进行标记。
  • 对于无法识别域名的路由器，可以忽略或留空。

步骤4总结：本节详细介绍了如何使用 traceroute 工具来分析互联网路径，并通过绘图和标记路由器的方法来深入理解网络路由的实际运作。

步骤5：IP头部校验和 (Step 5: IP Header Checksum)

目标 (Objective)

  • 学习如何通过验证数据包来计算IP头部校验和。校验和算法通过16位一次性加法计算，并且整个IP头重新计算的总和将为零。

校验和计算步骤 (Checksum Calculation Steps)

  1. 将IP头部分为10个16位（两个字节）的词，每个词在Wireshark的数据包数据面板底部以4个十六进制数字显示。
  2. 使用常规加法将这10个词相加。可以使用十六进制计算器，或者将它们转换为十进制，相加后再转换回十六进制。
  3. 计算1的补码总和。如果你的加法结果超出了4个数字的字词大小，将超出的部分加回到余数中。例如，如果结果是5a432，你会加上5得到a437。

结果分析 (Result Analysis)

  • 最终结果应该是0xffff。这实际上在1的补码形式中表示0。

注意事项 (Points to Note)

  • 如果计算的总和不正确，可以在Wireshark中检查它是否标记为“正确”。
  • 如果校验和计算出来不是你所期望的值，并且你确信校验和必须是错误的，可以调整Wireshark的设置来验证头部校验和。

步骤5总结：本节介绍了如何计算和验证IP头部的校验和，这对于理解IP通信的完整性和数据包验证至关重要。
