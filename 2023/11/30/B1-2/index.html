COM410 编程实践 - B1.2 计数操作（COM410 Programming in Practice - B1.2 Counting Operations）

总结：这是奥尔斯特大学（Ulster University）的COM410课程的一部分，专注于编程实践中的计数操作。

算法分析（Analysis of Algorithms）

  • 计算算法的实际时间需求是困难的（Difficult to compute the actual time requirement of an algorithm）
  • 寻找一个与问题规模相关且能反映算法实际时间需求的函数（Instead, find a function of the problem size that behaves like the algorithm’s actual time requirement）
    • 当时间需求增加某个因子时，函数的值以相同的因子增加，反之亦然（As the time requirement increases by some factor, the value of the function increases by the same factor and vice versa）
    • 这称为增长率函数，因为它衡量了算法时间需求随问题规模增长的情况（This is known as a growth-rate function）

总结：算法分析涉及寻找一个能反映算法时间效率的增长率函数，这有助于理解算法性能随问题规模的增长趋势。

计算基本操作（Counting Basic Operations）

  • 算法的基本操作是其总时间需求的最重要贡献者（An algorithm’s basic operation is the most significant contributor to its total time requirement）

  • 基本操作不一定是最频繁的操作（The most frequent operation is not necessarily the basic operation）

    // 算法 A
long sum = 0;
for (long i=1; i<=n; i++) {
    sum = sum + i;  // 基本操作标记为加法
}

// 算法 C
long sum = 0;
sum = n * (n + 1) / 2;  // 基本操作标记为乘法和除法

算法性能分析（Algorithm Performance Analysis）

基本操作计数（Counting Basic Operations）

  • 忽略非基本操作（如变量初始化、控制循环等）不会影响对算法速度的最终结论。

算法A（Algorithm A）

  • 基本操作：sum = sum + i;
  • 操作次数：2n + 2
  • 时间复杂度：线性增长，与n成正比。
总结

算法A的时间复杂度是线性的，这意味着算法的运行时间随着输入大小n的增加而线性增加。对于较大的数据集，算法A的效率可能不如具有更低时间复杂度的算法。

算法B（Algorithm B）

  • 基本操作：sum = sum + i;
  • 操作次数：n * (n + 1) / 2
  • 时间复杂度：二次增长，与n^2成正比。
总结

算法B的时间复杂度是二次的，这表明运行时间会随着输入大小的平方增长。这种类型的算法可能在小规模数据集上表现良好，但在大规模数据集上效率低下。

算法C（Algorithm C）

  • 基本操作：sum = n * (n + 1) / 2;
  • 操作次数：3
  • 时间复杂度：常数时间，与n无关。
总结

算法C有一个常数的时间复杂度，这意味着无论输入大小n如何，它执行的操作数量都是固定的。这种算法通常是最优的，因为它能在可预测的时间内完成任务，不受输入大小影响。

估算性能 (Estimating Performance)

  • 考虑两种算法，分别需要 10n 操作和 n^3 操作。
    • 算法A：执行 10n 操作。
    • 算法B：执行 n^3 操作。
  • 询问这两种算法哪一种在执行操作方面最慢。

算法比较

  • 算法A (10n)

    • 当 n 较小的时候，算法可能看似高效。
    • 随着 n 的增大，所需执行的操作数线性增加。

  • 算法B (n^3)

    • 初始时对于小的 n 值，可能执行操作不多。
    • 但是随着 n 的增加，所需操作数的增长速率远超算法A，呈现出立方增长的趋势。

总结

在两种算法中，算法B（执行 n^3 操作）随着 n 的增大，所需的操作数量呈指数级增加，因此在执行操作方面比算法A（执行 10n 操作）更慢。这反映了在算法性能评估中，复杂度的增长速度对于大规模数据处理的影响是非常关键的因素。

最好、最坏和平均情况 (Best, Worst, and Average Cases)

  • 对于某些算法，执行时间仅取决于数据集的大小。
  • 对于其他算法，执行时间可能取决于数据集的大小和性质。例如，在整数数组中查找特定整数值的场景：
    • 如果算法在数组的第一个元素中找到了值，它仅进行一次比较 — 这是最好情况。
    • 如果算法在遍历每个数组元素后找到值 — 这是最坏情况。

最好情况和最坏情况的分析

  • 最好情况 (Best Case)

    • 算法在最理想的情况下的执行效率。
    • 例如，数组排序算法在已经排好序的数组上的执行效率。

  • 最坏情况 (Worst Case)

    • 算法在最不理想的情况下的执行效率。
    • 例如，搜索算法在必须遍历整个数组才能找到目标值的情况。

平均情况的考虑

  • 我们通常考虑平均情况来评估算法的性能。
  • 平均情况的估算比最好情况和最坏情况更困难，它并不是最好情况和最坏情况的简单平均。

总结

在算法性能分析中，我们不仅需要考虑最好情况和最坏情况，更重要的是考虑平均情况，因为它提供了一个更加全面和实际的性能评估。这种评估可以帮助我们更好地理解算法在处理不同数据集时的表现。

  • 参数值分别为 1, 10, 100, 1000, 10000, 100000 和 1000000。

结果输出

  • 将执行时间的测量结果打印在表格中。
  • 观察每个方法随参数值增加的执行时间增长率。
