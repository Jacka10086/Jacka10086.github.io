用户目标分解（User goal decomposition）

  • 分解用户目标

    • 用户目标分解是将用户的主要目标细化为更小、更具体的子目标的过程。
    • 这有助于理解和组织实现主要用户目标所需的步骤。

  • 主要目标与子目标

    • 主要目标：作为不需要的物品的拥有者，我可以在在线拍卖网站上注册，以便我能出售我的物品。
    • 子目标1：作为不需要的物品的拥有者，我可以输入我的注册信息（姓名、电子邮件等），以便我能在在线拍卖网站上注册。
    • 子目标2：作为不需要的物品的拥有者，我可以提交我的注册信息，以便我能在在线拍卖网站上注册。

总结

用户目标分解帮助设计和开发团队清晰地识别和组织用户实现其主要目标所需要完成的具体步骤。它通过将大目标拆解成更小的、可操作的任务，来简化用户的任务流程，并优化整个用户体验。

三个命名的目标层次（Three named goal levels）

  • 概要层次（Summary Level）

    • 也称为云/风筝层次（Cloud/Kite Level），通常用白色表示。
    • 这个层次概括了最高级的目标或愿景，相当于从高层次全局视角观察目标。

  • 用户目标层次（User Goal Level）

    • 也称为海平面层次（Sea Level），通常用蓝色表示。
    • 在这个层次上，目标被定义为用户在使用产品或服务时想要达成的具体目标。

  • 子功能层次（Subfunction Level）

    • 也称为水下层次（Underwater Level），通常用靛蓝色表示。
    • 这个层次包含了实现用户目标所需的更具体的操作或任务。

总结

这三个层次共同构成了目标设置的框架，帮助团队从不同高度理解用户目标，并识别实现这些目标所需的具体任务和功能。从高层的概要到具体的子功能，每个层次都对应着不同深度的目标和任务细节。

分离非功能性需求 (Split out NFRs)

功能性与非功能性需求的区分

  • 功能性需求指的是系统必须做什么，即具体的功能点。
  • 非功能性需求则描述了系统如何运行，以及系统运行的质量标准。

推迟的非功能性需求 (Defer the following NFRs)

  • 某些NFRs可能在项目的早期阶段不予考虑，待到系统框架建立后再进行详细规划和实施。

性能 - 速度 (Performance - speed)

  • 描述系统响应请求的速度快慢。

可伸缩性 - 大数据或大量用户 (Scalability - large data or large set of users)

  • 系统能否在数据量或用户数增加时保持效能。

并发性 - 竞态条件、数据锁定、并发用户 (Concurrency - Race conditions, data locking, concurrent users)

  • 多个进程或线程同时访问数据资源时系统的表现。

可用性/可靠性 - 正常运行时间、稳固性 (Availability/Reliability - uptime, robustness)

  • 系统持续运行不中断的能力，以及在出现问题时保持功能的能力。

安全性 - 防止攻击 (Security - protect from attack)

  • 保护系统不受恶意攻击和未授权访问。

可用性/可访问性 - 对每个人来说都容易使用 (Usability / accessibility - easy to use for everyone)

  • 系统的界面设计是否直观，以及是否支持残疾人士等所有用户群体。

用户体验 - 界面美观 (UX - making it look nice)

  • 界面设计的美观程度，对用户体验的影响。

跨浏览器/平台 - 支持多种客户端 (Cross browser / platform - work for a variety of clients)

  • 系统是否支持不同的浏览器和操作系统。

国际化 - 语言支持 (Internationalisation - language support)

  • 系统是否支持多种语言，适应不同地区用户的需求。

总结

非功能性需求是确保软件系统具有良好性能、扩展性、安全性和用户体验的关键因素。它们通常不会在项目初期全部实施，而是根据项目进度逐步加以考虑和优化。

按用户界面通道分离 (Split by User Interface Channel)

跨浏览器/平台支持的特殊情况 (Special case for cross browser / platform support)

  • 考虑到软件需要在不同的浏览器和操作系统上运行，需要特别注意其兼容性和适应性。

多种设备支持 (Multiple devices)

  • **桌面 (Desktop)**：软件需适应各种分辨率和操作系统的桌面环境。
  • **平板 (Tablet)**：软件需优化触控操作和屏幕尺寸适配。
  • **移动 (Mobile)**：软件需适应不同尺寸的移动设备，并考虑移动数据的使用效率。

支持的平台 (Platforms)

  • Windows：软件需适配Windows操作系统的各个版本。
  • Mac：软件需适配MacOS操作系统，考虑其独特的用户界面设计标准。
  • iOS：软件需适配iPhone和iPad等设备，考虑App Store的发布要求。
  • Android：软件需适配Android系统的多种设备和版本。

总结

为了确保软件产品能够覆盖更广泛的用户群体，开发时需考虑多种用户界面通道，包括不同的设备类型和操作系统平台。这样可以提高产品的可访问性和用户满意度。

按用户类型分离 (Split by User Type)

用户故事的多样性 (Some stories have a diverse range of users)

  • 用户故事可能涉及多种不同类型的用户，每个用户类型都有其独特的需求和故事。

用户注册案例 (As a user I want to register)

  • 用户注册功能需要考虑到不同用户角色的需求和流程。

教师 (Teacher)

  • 注册流程可能需要验证教师资格和学校信息。

家长 (Parent)

  • 注册流程应简单明了，方便家长快速完成。

校长 (Principle)

  • 校长的注册流程可能需要高级别的访问权限和审核流程。

总结

根据用户类型的不同，注册故事和需求会有所不同。系统设计时需要考虑这些差异，确保每个用户角色都有适合其需求的注册流程。

将总结性目标分解为用户目标 (Split summary goals into user goals)

总结性目标层面 (Summary level)

  • 概述性的用户目标提供了一个宽泛的视角，概括了用户希望通过使用系统达成的综合性目标。

管理文章 (As blogger I can manage my article)

  • 这表明博客作者希望有能力对其文章进行总体管理，包括创建、查看、编辑和删除等操作。

用户目标层面 (User goal level)

  • 用户目标层面则是将总结性目标细分为具体可执行的任务，每个任务对应用户的一种操作。

创建文章 (As a blogger I can create my article)

  • 用户需要能够创作并提交新的博客文章。

查看文章 (As a blogger I can view my article)

  • 用户需要能够查看已发布或草稿状态的博客文章。

编辑文章 (As a blogger I can edit my article)

  • 用户需要能够对已发布或草稿状态的博客文章进行修改。

删除文章 (As a blogger I can delete my article)

  • 用户需要能够移除不再需要的博客文章。

图表解释

  • 图表展示了从一个汇总目标（如文章管理）到分解的用户目标（如创建、查看、编辑和删除文章）的关系。
  • 上层的黄色块代表了总结性目标，而下层的四个黄色块分别代表了具体的用户目标。
  • 此分解过程帮助明确每个目标对应的用户故事和使用案例，为功能开发提供清晰的指导。

总结

将总结性目标细分为具体的用户目标，有助于软件开发团队理解和实现用户的具体需求。通过这种方式，每个小的目标都对应用户的一个具体操作，使得功能设计和实现更加明确和针对性。

按场景/流程分离 (Split by Scenario / Flow)

正常路径与非正常路径 (Happy paths versus unhappy paths)

  • 正常路径：用户成功完成期望操作的流程。
  • 非正常路径：用户在操作过程中遇到错误或阻碍的流程。

提供常见的正常路径 (Deliver common happy paths)

  • 确定并优化用户在使用软件时最可能遵循的成功路径。

识别替代流程 (Identify alternative flows)

  • 探索并定义当正常路径不可用时的备选操作步骤。

登录场景 (Sign in)

  • 描述用户登录过程中可能遇到的不同结果。

成功 (success)

  • 用户输入正确的用户名和密码后成功登录。

用户名或密码无效 (Invalid username or password)

  • 用户在登录时输入了错误的凭据。

尝试失败3次 (failed 3 times)

  • 用户连续三次登录失败，可能因为忘记凭据。

账户被锁定 (account locked)

  • 多次登录尝试失败后，系统为了安全考虑自动锁定账户。

图表解释

  • 图表显示了在设计用户登录功能时，需要考虑的不同场景和流程。
  • 图表将正常路径与可能的非正常路径进行了区分，并展示了在非正常路径下可能的不同情况。

总结

在软件开发过程中，识别并设计不同的用户场景和流程对于确保软件的健壮性和用户友好性至关重要。这包括处理成功的登录尝试（正常路径），以及错误的登录信息或账户锁定等问题（非正常路径）。

按场景/流程分离 (Split by Scenario / Flow)

核心正常路径 - 用户登录 (Write out steps for core happy path - sign in)

  • 描述用户在理想情况下进行登录操作的步骤。
  1. 用户请求登录 (User requests sign in)
    • 用户开始登录流程。
  2. 系统提供登录表单 (System present sign in form)
    • 系统展示登录界面供用户输入凭据。
  3. 用户输入用户名和密码 (User enters username and password)
    • 用户在指定区域输入登录信息。
  4. 系统验证登录信息 (System verifies sign in details for an existing account)
    • 系统检查输入的用户名和密码是否匹配。
  5. 系统确认账户有效 (System verifies account is active)
    • 系统检查账户是否处于激活状态。
  6. 系统确认账户未被锁定 (System verifies account is not locked)
    • 系统检查账户是否因多次登录失败而被锁。
  7. 系统登录并显示仪表板 (System logs in and displays dashboard)
    • 登录成功后，系统展示用户仪表板。

替代路径 (Alternative paths)

  • 当核心正常路径的某个步骤失败时，系统应提供替代路径，确保用户可以恢复并完成登录。
  1. 用户名或密码输入错误 (Invalid username or password)
    • 系统提供错误提示，并允许用户重试。
  2. 连续登录失败 (Failed 3 times)
    • 系统提供密码重置或联系支持的选项。
  3. 账户被锁定 (Account locked)
    • 系统提供账户解锁流程的指引。

图表解释

  • 图表详细列出了用户登录的标准流程，以及在登录失败时可能遵循的替代路径。
  • 通过这种分解，可以确保系统设计能够处理各种登录场景，包括用户错误和账户问题。

总结

理解并设计用户登录的核心正常路径及其替代路径对于提供无缝的用户体验至关重要。这有助于确保在用户遇到问题时，系统能够提供合适的解决方案，维持用户的使用流畅性和满意度。

青铜镀层/黄金镀层流程 (Bronze plated / Gold plated flow)

基础但足以满足最小可行产品 (Basic but good enough for MVP)

  • 基本需求：在最小可行产品(MVP)阶段，功能要求简单但足够使用。
    • 使用文本字段来收集必要的信息，例如出生日期。

额外的功能和装饰 (Bells and whistles)

  • 增强体验：在产品成熟后，可以添加更多额外的功能和装饰来提升用户体验。
    • 如使用日历选择器来输入日期，提供更直观的操作。

用户故事 (User story)

  • 作为博客作者，我希望注册账户：这是用户希望通过系统实现的目标。

必填信息 (Required information)

  • 出生日期是必填项：明确标出必填的信息，保证用户提供所需的详细数据。

图表解释

  • 图表通过不同的奖杯图标展示了产品开发的两个阶段。
  • 青铜奖杯代表基础功能，足以支持产品的最初版本。
  • 金奖杯代表增加的额外功能，可在产品功能完善后添加。

总结

在软件开发中，区分青铜镀层和黄金镀层流程有助于聚焦于核心功能的快速发布，并在后续阶段根据用户反馈和市场需求增加额外功能。这种策略确保了产品能够快速上市，并且随着时间的推移逐渐完善。

青铜镀层技巧 (Bronze plating tricks)

在软件开发的初期阶段，通常会采取一些策略来简化开发过程，这些策略被称为“青铜镀层技巧”。以下是这些技巧的分类及解释：

手动与自动化 (Manual versus automatic)

  • 对于低频任务选择手动操作：在任务不经常执行时，手动操作可能更经济，因为自动化的成本和复杂性可能不值得。

硬编码与可配置 (Hard-coded versus configurable)

  • 选择硬编码而非可配置：对于不太可能改变的参数，硬编码可以减少初期的复杂性。

假设与询问 (Assumed versus asked)

  • 默认设置对比个性化选项：提供默认选项可以减少用户决策负担，简化使用流程。

简单与复杂 (Simple versus complex)

  • 选择直接输入对比复杂查询：当简单的解决方案足够好时，没有必要构建复杂的系统。

图表解释

  • 图表列出了在最小可行产品（MVP）阶段可能采取的不同技巧。
  • 通过对比不同的策略，可以帮助决策者在功能实现和资源投入之间做出权衡。

总结

这些“青铜镀层技巧”有助于团队在产品初期快速推进，减少开发时间和成本。它们使团队能够集中精力于实现核心功能，而不是花费时间在那些可能在产品后期才需要的高级特性上。

骨架 (Skeleton)

在软件开发过程中，”骨架”方法强调先构建一个端到端的基础结构，然后再补充完善细节。

构建端到端基础框架 (Build end to end and then fill in the gaps)

  • 从注册按钮到提交表单：首先设置一个可以触发注册流程的按钮，然后创建一个空表单让用户填写，最后添加一个提交按钮完成注册过程。
  • 集中在用户界面流程：在这个阶段，先模拟出整个用户界面流程，不涉及后端逻辑的详细实现。

补充细节 (Stub out the back end code to focus on UI flow)

  • 在UI流程确定之后，再逐步实现后端逻辑，填补前端UI与后端逻辑之间的空白。

图表解释

  • 图表通过鱼骨图的形象表示了骨架方法的概念，骨架即是整个产品的基础结构。
  • 鱼骨图中的每一根骨头都代表开发过程中需要补充完善的部分。

总结

“骨架”方法有助于快速构建产品的核心框架，并在此基础上逐步完善和填充细节，这样可以确保在开发过程中始终聚焦于用户体验的连贯性。
