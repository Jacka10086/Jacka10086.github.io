网站测试 (Website Testing)

关键考虑因素针对网页开发者。

功能测试 (Functionality Testing)

  • 确保所有功能稳健运作。
  • 涵盖多种功能类型，如表单、Cookies以及数据库完整性。
  • 确保测试用例涵盖边界条件。

安全测试 (Security Testing)

  • 对于网站至关重要。
  • 测试机器人活动检测。
  • 防止SQL脚本注入。

集成测试 (Integration Testing)

  • 测试所有系统组件是否能够正常协同工作。
  • 在团队开发环境中尤其重要。

性能测试 (Performance Testing)

  • 帮助确定产品在不同使用情况下的性能。
  • 包括压力测试、可伸缩性测试和负载测试。

兼容性测试 (Compatibility Testing)

  • 网络应用程序的兼容性至关重要。
  • 须符合标准规范和法律合规性。

可用性测试 (Usability Testing)

  • 网站应符合可用性标准。
  • 考虑认知负荷、支持用户目标、信息设计、导航设计以及视觉设计。
  • 包括所有用户的测试，例如无障碍测试。

总结：以上内容详细介绍了网站测试的不同方面，旨在确保网站在多方面都能达到预定的标准和用户期望。

网站测试 (Website testing)

  • 关键考虑因素：为Web开发人员提供关于如何测试网站的重要提示和方法。
  • 总结：为网站开发的测试提供了一份指南，强调了测试在整个开发过程中的重要性。

测试所有方面 (Testing all aspects)

  • 功能测试 (Functionality testing)
  • 安全测试 (Security testing)
  • 集成测试 (Integration testing)
  • 性能测试 (Performance testing)
  • 兼容性测试 (Compatibility testing)
  • 可用性测试 (Usability testing)
  • 总结：涵盖了网站测试的不同方面，强调了为确保高质量和用户满意度必须进行全面测试。

功能性测试 (Functionality Testing)

  • 确保所有功能鲁棒性运作
  • 涉及多种功能类型，例如表单、Cookies和数据库完整性
  • 测试案例需覆盖边界条件
  • 总结：提出了确保网站所有功能都能正常运作的测试方法，以及测试需要覆盖的要点。

安全测试 (Security Testing)

  • 对于网站来说至关重要
  • 测试机器人活动检测
  • 检查避免直接访问服务器文件结构
  • 防止SQL脚本注入
  • 总结：概述了确保网站安全的各种测试，包括预防恶意攻击和数据泄露。

集成测试 (Integration testing)

  • 测试所有系统组件是否正常协同工作
  • 重要性在于团队开发环境中的应用
  • 总结：说明了测试不同系统组件如何整合在一起工作的重要性，以及这在团队合作中的作用。

性能测试 (Performance Testing)

  • 帮助确定产品在不同用例下的性能
  • 包括压力测试、可伸缩性测试和负载测试
  • 总结：阐述了如何测试产品在不同条件下的性能表现，并强调了多平台测试的必要性。

兼容性测试 (Compatibility Testing)

  • 您的Web应用程序的兼容性至关重要
  • 遵守标准、可访问性和法律合规性
  • 包括浏览器兼容性和操作系统兼容性
  • 总结：强调了测试网站在不同浏览器和操作系统上的兼容性的重要性。

可用性测试 (Usability Testing)

  • 网站应遵守可用性标准
  • 考虑认知负荷、用户目标支持、信息设计、导航设计和视觉设计
  • 包括所有用户，即无障碍性测试
  • 总结：提出了确保网站符合用户友好性标准的测试方面，包括设计和可访问性考虑。

敏捷Scrum指标 / KPIs（Agile Scrum Metrics / KPIs）

  • Sprint目标成功（Sprint Goal Success）
    • 选择冲刺目标（Select a sprint goal）
    • 运行冲刺（Run the sprint）
    • 验证目标是否达成（Validate if the goal is met）

总结：敏捷Scrum的指标或关键绩效指标（KPIs）是用来评估Scrum团队在冲刺期间是否成功实现目标的工具，它们包括选择目标、运行冲刺和验证目标的实现情况。

Sprint目标成功（Sprint Goal Success）

  • 选择冲刺目标（Select a sprint goal）
  • 运行冲刺（Run the sprint）
  • 验证目标是否达成（Validate if the goal is met）

总结：实现Sprint目标是衡量Scrum团队成功的关键因素。这个过程从选择一个具体目标开始，然后执行冲刺，并在结束时验证目标是否已经达成。

逃逸缺陷（Escaped Defects）

  • 定义：逃逸缺陷是在软件发布后才被用户发现的缺陷。
  • 图表说明：
    • 中心圆圈表示代码库，外围圆圈表示生产环境。
    • 从代码库到生产的箭头显示了缺陷是如何“逃逸”出去的。
    • 用户在外围圆圈，表示他们与生产环境交互并可能遇到这些缺陷。

总结：逃逸缺陷图表强调了在软件开发生命周期中确保质量的重要性。它说明了缺陷管理的重要性，以及为何需要在早期发现并修复缺陷，避免它们影响到最终用户。

Team Velocity（团队速度）

  • 团队速度是衡量敏捷团队在一定周期内完成工作量的指标。
    • 通常以故事点（Story Points）来量化。
    • 图表显示了连续的若干个迭代（Sprints）中团队速度的变化。
  • 图中展示了一个警告标志，提醒我们“不要将团队速度作为目标或用于比较团队”。
    • 这是因为过度关注速度可能会导致团队牺牲质量以追求速度。

图表解释

  • X轴代表迭代编号，从1到10。
  • Y轴代表故事点数，是衡量工作量的单位。
  • 折线图展示了每次迭代团队完成的故事点数。
    • 一般希望看到的是稳定或递增的趋势，这表示团队效率在提高。

总结

此图表是评估敏捷团队效率的一种工具。它有助于团队和组织了解每次迭代能够完成的工作量，从而对未来迭代进行更好的计划和预测。然而，重要的是不要将速度作为团队比较的唯一或主要指标，因为这可能会激励错误的行为，如牺牲质量以提高速度。

Sprint Burn Down（冲刺燃尽图）

  • 冲刺燃尽图用于跟踪冲刺中工作进度。
    • 目的是确保团队能在冲刺结束前完成目标。
    • 图表展示了剩余工作量随时间递减的情况。

图表解释

  • X轴代表时间，通常表示冲刺中的天数。
  • Y轴代表剩余的故事点数，表示还需完成的工作量。
  • 蓝色条形图显示每天结束时剩余的故事点数。
  • 红色线条代表理想的燃尽速度，即每天应该完成的工作量。
    • 如果蓝色条形图低于红线，意味着团队超前于计划。
    • 如果蓝色条形图高于红线，意味着团队落后于计划。

总结

冲刺燃尽图是敏捷团队用来确保按时交付的关键工具。通过它，团队可以清晰地看到每天的工作进度，以及与计划的对比。这有助于团队成员调整工作节奏和策略，确保冲刺目标的实现。图表的实时更新和可视化使得团队能够及时发现问题并进行调整，以避免在冲刺结束时才发现大量的未完成工作。

Agile Scrum Metrics / KPIs（敏捷Scrum指标/KPIs）

Sprint Goal Success（冲刺目标成功）

  • 选择冲刺目标
  • 运行冲刺
  • 验证目标是否达成
  • 总结：冲刺目标的成功取决于明确的目标设定、执行过程以及最终的成果验证。

Escaped Defects（逃逸缺陷）

  • 从代码库到生产环境中的缺陷
  • 用户发现的问题
  • 总结：逃逸缺陷是衡量生产软件质量的一个关键指标，反映了从代码库到用户手中软件的缺陷情况。

Team Velocity（团队速度）

  • 以故事点计量的团队速度
  • 不应作为目标或用于比较团队
  • 总结：团队速度是衡量团队在冲刺中交付价值的速率，不推荐作为固定目标或比较不同团队的基准。

Sprint Burn Down（冲刺燃尽图）

  • 每天完成的故事点数
  • 剩余的工作量
  • 预估完成日期
  • 总结：冲刺燃尽图显示了冲刺过程中工作量的减少和团队完成任务的速率。

Measuring Business Goals（衡量业务目标）

  • 上市时间
  • 投资回报率（ROI）
  • 资本再部署
  • 客户满意度
  • 总结：业务目标的衡量包括多个维度，旨在评估产品上市效率、投资回报、资本使用效率和客户满意度。

Scrum Metrics（Scrum指标）

  • 每日Scrum和冲刺回顾
  • 团队满意度
  • 团队成员流动率
  • 总结：Scrum指标关注于团队的日常运作、成员满意度以及人员稳定性，是保障团队健康运作的关键因素。

Scrum Reporting（Scrum报告）

  • 冲刺和发布燃尽图 - 进度概览
  • 冲刺速度 - 价值交付回顾
  • 范围变更 - 发布期间添加的故事
  • 团队容量 - 全职开发者数量？请假情况？参与其他项目？
  • 逃逸缺陷 - 生产软件的状况
  • 总结：Scrum报告提供了项目进度、团队表现和软件质量的全面视图，帮助团队和利益相关者理解项目状态。

敏捷Scrum度量指标 / KPIs

这部分内容提供了关于敏捷Scrum度量指标的概览。主要介绍了在Scrum实践中，哪些关键绩效指标（KPIs）是重要的。

Sprint Goal Success

  • 选择Sprint目标：团队需要确定一个清晰的目标，以便在Sprint中追求。
  • 运行Sprint：在Sprint期间，团队致力于实现设定的目标。
  • 验证目标是否达成：在Sprint结束时，需要检查目标是否已经实现，以评估Sprint的成功与否。

故障逃逸

  • 展示了代码生产过程中可能出现的问题，并且这些问题是如何在生产环境中被用户发现的。

Team Velocity

  • 故事点：以故事点来衡量团队在连续Sprints中的工作量。
  • Sprints：表示时间序列上的Sprint迭代次数。
  • 警告：不应将团队速度作为比较团队间差异的目标。

Sprint Burn Down

  • 故事点：任务完成的量化表示。
  • 天数：Sprint周期内的时间流逝。
  • Burndown速度：展示了任务完成的速率。

测量商业目标

  • 上市时间：产品开发到推向市场所需的时间长度。
  • ROI：投资回报率。
  • 资本再部署：在产品生命周期中重新分配资本的能力。
  • 客户满意度：衡量客户对产品或服务满意程度的指标。

Scrum度量

  • 每日Scrum和Sprint回顾：衡量团队沟通和反思效率的指标。
  • 团队满意度：团队成员对工作环境的满意程度。
  • 团队成员流失率：团队成员离职的频率。

Scrum报告

  • Sprint和发布倒数：展示进度的总览。
  • Sprint速度：随时间推移交付价值的回顾。
  • 范围变化：在发布过程中增加的故事数量。
  • 团队容量：全职开发者的数量？假期？病假？参与其他项目？
  • 逃逸缺陷：生产软件的质量如何？

通过上述内容的概述，我们可以了解到敏捷Scrum实践中各种度量指标和KPIs对于项目管理的重要性。这些指标不仅帮助项目团队监控进度，还能够评估产品的商业价值和客户满意度。

总结：冲刺燃尽图帮助团队追踪进度，确保按时完成迭代目标。

Measuring Business Goals (衡量业务目标)

  • 目标包括：
    • 上市时间
    • 投资回报率（ROI）
    • 资本再部署
    • 客户满意度

总结：企业应以多维度指标来衡量其业务目标的实现情况，这对于指导决策和改进策略至关重要。

Time to Market (上市时间)

  • 描述了从开发团队开始工作，通过迭代（Sprints）最终实现产品发布，并产生收益的过程。
  • 图表强调了价值传递的流程，从内部开发到最终用户。

总结：快速上市是企业成功的关键，它可以加速收入流并提升市场竞争力。

ROI (投资回报率)

  • 对比了运行迭代的成本与产品收入。
  • 重点是理解成本和收益之间的关系。

总结：ROI是评估项目财务效益的重要指标，有助于企业决定投资哪些项目。

合并拉取请求（Merged Pull Request）

  • 分支（Branch）

    • Fork： 从主分支创建一个分支，开始独立开发。
    • Main Branch： 主分支，所有功能最终合并到此。

  • 开发过程（Development Process）

    • First commit： 在分支上的第一个提交，标志着功能开发的开始。
    • Commits： 随着开发的进行，会有多次提交，表示代码的增量改变。

  • 代码审查（Code Review）

    • Pull request opened： 开发完成后，创建拉取请求（Pull Request）以请求合并到主分支。
    • Comments： 在拉取请求中，其他团队成员可以进行代码审查，留下评论和建议。

  • 合并（Merging）

    • Pull request merged： 审查完成后，如果代码符合标准，则拉取请求被合并到主分支。

总结

本图表显示了代码从分支到合并入主分支的典型流程。这个过程涵盖了代码的提交、审查和合并操作，以确保代码质量并促进团队协作。合并拉取请求是代码集成和版本控制的关键环节，确保了代码的可维护性和团队的协作效率。

  • 图表解释

    • 横轴（Development Time）：展示了开发时间的流逝。
    • 纵轴（Code Churn）：指的是在开发周期内代码变动的频率和规模。

  • 数据点分析

    • 曲线：表示代码变动随时间的增加而发生的波动情况。
    • 曲线上升：意味着代码变动的频率和规模在增加。

  • 图表关键点

    • 晚期代码波动（Late Churn）： 红色虚线之后的曲线上升区域，这通常指的是开发晚期还有大量的代码变动，这可能是由于重新设计、优化或修复在晚期测试中发现的问题。

总结

该图表揭示了在开发过程的后期，代码变动的频率和规模有显著增加，这通常不是一个好现象，因为它可能意味着项目需求不明确、计划不充分或者存在其他问题。团队应该探究晚期代码波动的原因，并寻求在项目早期阶段解决问题，以减少晚期的波动，确保开发流程的顺利和产品质量。
